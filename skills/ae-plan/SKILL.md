# 创建技术计划

**注意：当前年份是 2026 年。** 在计划日期和搜索最新文档时使用此年份。

`ae:brainstorm` 定义**做什么**。`ae:plan` 定义**怎么做**。`ae:work` 执行计划。之前的头脑风暴是有用的上下文但从不强制要求——`ae:plan` 可以从任何输入工作：需求文档、Bug 报告、功能想法，或粗略描述。

**当被直接调用时，始终进行计划。** 永远不要将直接调用分类为"不是计划任务"而放弃工作流。如果输入不清晰，提出澄清问题或使用计划引导（Phase 0.4）建立足够的上下文——但始终留在计划工作流中。

本工作流产生一份持久的实现计划。它**不会**实现代码、运行测试，或从执行时结果中学习。如果答案依赖于修改代码并观察结果，那属于 `ae:work`，不在这里。

## 交互方式

当可用时，使用平台的 `question` 工具。当向用户提问时，优先使用平台的阻塞式提问工具。否则，在聊天中呈现编号选项并等待用户回复后再继续。

每次只问一个问题。当存在自然选项时，优先使用简洁的单选选择。

## 功能描述

<feature_description> #$ARGUMENTS </feature_description>

**如果上面的功能描述为空，询问用户：** "你想计划什么？描述你心中的任务、目标或项目。" 然后等待他们的回复再继续。

如果输入存在但不清晰或规格不足，不要放弃——问一到两个澄清问题，或进入 Phase 0.4 的计划引导建立足够的上下文。目标始终是帮助用户计划，永远不是退出工作流。

**重要：计划文档中的所有文件引用必须使用仓库相对路径（例如 `src/models/user.ts`），永远不要使用绝对路径（例如 `/Users/name/Code/project/src/models/user.ts`）。这适用于所有位置——实现单元文件列表、模式引用、源文档链接和正文提及。绝对路径会破坏跨机器、工作树和团队成员的可移植性。**

## 核心原则

1. **以需求为真相来源** - 如果 `ae:brainstorm` 产生了需求文档，计划应基于它构建，而不是重新发明行为。
2. **决策，不是代码** - 捕获方法、边界、文件、依赖、风险和测试场景。不要预先编写实现代码或 Shell 命令编排。伪代码草图或 DSL 语法有助于审阅者验证方向时是欢迎的——但必须明确标注为方向性指导，而非实现规范。
3. **先研究再结构化** - 在最终确定计划之前，根据需要探索代码库、制度经验和外部指导。
4. **合适的产物规模** - 小工作获得紧凑计划。大工作获得更多结构。理念在每个深度都保持一致。
5. **将计划与执行发现分离** - 在此解决计划时的问题。明确将执行时的未知事项推迟到实现阶段。
6. **保持计划可移植** - 计划应作为活文档、审阅产物或 Issue 正文使用，而不嵌入工具特定的执行器指令。
7. **在需要时轻量标注执行姿态** - 如果请求、源文档或仓库上下文明确暗示了测试优先、特征化测试优先或其他非默认执行姿态，在计划中将其反映为轻量信号。不要将计划变成逐步执行编排。

## 计划质量标准

每个计划应包含：
- 清晰的问题框架和范围边界
- 可追溯到请求或源文档的具体需求
- 所提议工作的仓库相对文件路径（绝不使用绝对路径——见计划规则）
- 包含功能的实现单元的显式测试文件路径
- 带有理由的决策，而不仅是任务
- 需要遵循的现有模式或代码引用
- 每个包含功能的单元的枚举测试场景，足够具体，让实现者清楚知道要测试什么，无需自行设计覆盖范围
- 清晰的依赖和排序

当实现者可以自信地开始工作而不需要计划替他们写代码时，计划就准备好了。

## 工作流

### Phase 0：恢复、来源和范围

#### 0.1 在适当时候恢复已有计划工作

如果用户引用了现有的计划文件，或在 `docs/ae/plans/` 中有明显的近期匹配计划：
- 读取它
- 确认是就地更新还是创建新计划
- 如果更新，保留已完成的复选框，仅修订仍然相关的部分

**深化意图：** 提及计划时使用的"深化"（或"deepening"）一词是深化快速路径的主要触发器。当用户说"深化计划"、"深化我的计划"、"运行一轮深化"或类似表述时，目标文档是 `docs/ae/plans/` 中的**计划**，而非需求文档。使用用户提供的任何路径、关键词或上下文来识别正确的计划。如果提供了路径，验证它确实是计划文档。如果匹配不明显，在继续之前向用户确认。

"加强"、"信心"、"空白"和"严谨"等词本身不足以触发深化。这些词出现在正常编辑请求中（"加强关于图表的那一节"、"测试场景有空白"），不应导致整体深化轮次。仅当请求明显针对整个计划且未指定要更改的特定章节或内容区域时，才将其视为深化意图——即使如此，也优先在进入深化流程之前向用户确认。

一旦计划被识别且看起来完整（所有主要章节存在、实现单元已定义、`status: active`）：
- 如果计划缺少 YAML frontmatter（非软件计划使用简单的 `# 标题` 标题加 `Created:` 日期而非 frontmatter），路由到 `references/universal-planning.md` 进行编辑或深化，而非 Phase 5.3。非软件计划不使用软件信心检查。
- 否则，短路到 Phase 5.3（信心检查和深化）的**交互模式**。这避免了重新运行完整的计划工作流，并让用户控制哪些发现被整合。

正常编辑请求（例如，"更新测试场景"、"添加新的实现单元"、"加强风险部分"）不应触发快速路径——它们遵循标准恢复流程。

如果计划已有 `deepened: YYYY-MM-DD` frontmatter 字段且用户未明确要求重新深化，快速路径仍应用相同的信心差距评估——它不会强制深化。

#### 0.1b 分类任务领域

如果任务涉及构建、修改或架构软件（引用代码、仓库、API、数据库，或要求构建/修改/部署），继续到 Phase 0.2。

如果任务是关于非软件领域且描述了值得计划的多步骤目标，读取 `references/universal-planning.md` 并遵循该工作流。跳过所有后续阶段。

如果真正模糊（例如，"计划一次迁移"但没有其他上下文），在路由之前询问用户。

对于其他所有情况（快速问题、错误消息、事实查询）**仅在自动选择时**，直接回复而不进入任何计划工作流。当被用户直接调用时，将输入视为计划请求——如需则提出澄清问题，但不退出工作流。

#### 0.2 查找上游需求文档

在提出计划问题之前，搜索 `docs/ae/brainstorms/` 中匹配 `*-requirements.md` 的文件。

**相关性标准：** 需求文档在以下情况下相关：
- 主题在语义上匹配功能描述
- 在过去 30 天内创建（如果文档明显仍然相关或明显过时，使用判断覆盖）
- 似乎涵盖相同的用户问题或范围

如果多个源文档匹配，询问使用哪一个。当平台的阻塞式提问工具可用时使用（见交互方式）。否则，在聊天中呈现编号选项并等待用户回复后再继续。

#### 0.3 将源文档作为主要输入

如果存在相关的需求文档：
1. 仔细阅读它
2. 宣布它将作为计划的源文档
3. 携带以下所有内容：
   - 问题框架
   - 需求和成功标准
   - 范围边界
   - 关键决策和理由
   - 依赖或假设
   - 待解决问题，保留它们是阻塞的还是推迟的
4. 使用源文档作为计划和研究的主要输入
5. 在计划中用 `(见源文档: <source-path>)` 引用重要的携带决策
6. 不要静默省略源内容——如果源文档讨论了某事，计划必须解决它，即使只是简短提及。在最终确定之前，扫描源文档的每个部分以验证没有内容被遗漏。

如果不存在相关需求文档，可以直接从用户的请求开始计划。

#### 0.4 计划引导（无需求文档或输入不清晰）

如果不存在相关需求文档，或输入需要更多结构：
- 评估请求是否已经足够清晰以直接进行技术计划——如果是，继续到 Phase 0.5
- 如果模糊主要是产品框架、用户行为或范围定义，推荐 `ae:brainstorm` 作为建议——但始终同时提供继续在此计划的选择
- 如果用户想在这里继续（或已经明确表示想要计划），运行下面的计划引导

计划引导应建立：
- 问题框架
- 预期行为
- 范围边界和明显的非目标
- 成功标准
- 阻塞问题或假设

保持引导简洁。它存在的目的是保留直接进入的便利性，而非替代完整的头脑风暴。

如果引导发现重大未解决的产品问题：
- 再次推荐 `ae:brainstorm`
- 如果用户仍想继续，在继续之前要求明确假设

如果引导发现有其他工作流能更好地服务用户：

- **有症状但无根因**（用户描述了异常行为但未识别原因）——宣布在计划之前需要调查，并引导用户使用调试方法。计划需要一个已知问题来解决；调试识别那个问题是什么。明确告知用户路由方向："这需要在计划之前进行调查。"
- **明确的任务可以执行**（已知根因，明显修复，无架构决策）——建议 `ae:work` 作为更快的替代方案，同时提供继续计划的选择。由用户决定。

#### 0.5 在计划之前分类待解决问题

如果源文档包含 `Resolve Before Planning` 或类似的阻塞问题：
- 在继续之前审查每一个
- **仅当**它是真正的技术、架构或研究问题时，将其重新分类为计划拥有的工作
- 如果它会改变产品行为、范围或成功标准，保持为阻塞项

如果存在真正的产品阻塞项：
- 明确提出它们
- 使用平台的阻塞式提问工具（可用时，见交互方式）询问用户是否：
  1. 恢复 `ae:brainstorm` 来解决它们
  2. 将它们转换为明确的假设或决策并继续
- 当真正的阻塞项未解决时不要继续计划

#### 0.6 评估计划深度

将工作分类为以下计划深度之一：

- **轻量级** - 小型、边界明确、低模糊性
- **标准** - 正常功能或有边界重构，有一些技术决策需要文档化
- **深度** - 跨切面、战略性、高风险或高模糊性的实现工作

如果深度不清晰，问一个针对性问题然后继续。

### Phase 1：收集上下文

#### 1.1 本地研究（始终运行）

准备简洁的计划上下文摘要（一到两段）作为研究代理的输入：
- 如果源文档存在，总结其中的问题框架、需求和关键决策
- 否则直接使用功能描述

并行运行这些代理：

- Task research:repo-research-analyst(范围: 技术、架构、模式。{计划上下文摘要})
- Task research:learnings-researcher(计划上下文摘要)

收集：
- 技术栈和版本（在 1.2 节中用于做出更精确的外部研究决策）
- 需要遵循的架构模式和约定
- 实现模式、相关文件、模块和测试
- AGENTS.md 中对计划有实质性影响的指导
- 来自 `docs/ae/solutions/` 的制度经验

#### 1.1b 检测执行姿态信号

决定计划是否应携带轻量的执行姿态信号。

寻找以下信号：
- 用户明确要求 TDD、测试优先或特征化测试优先的工作
- 源文档要求测试优先实现或遗留代码的探索性加固
- 本地研究显示目标区域是遗留的、测试薄弱的或历史上脆弱的，表明在更改行为之前需要特征化覆盖

当信号明确时，在相关实现单元中静默携带。

仅当姿态会实质性改变排序或风险且无法负责任地推断时才询问用户。

#### 1.2 决定外部研究

基于源文档、用户信号和本地发现，决定外部研究是否增加价值。

**读懂言外之意。** 注意对话中的信号：
- **用户熟悉度** — 他们是否指向特定文件或模式？他们可能很了解代码库。
- **用户意图** — 他们想要速度还是彻底性？探索还是执行？
- **主题风险** — 安全、支付、外部 API 无论用户信号如何都需要更多谨慎。
- **不确定性水平** — 方法是清晰的还是仍然开放的？

**利用 repo-research-analyst 的技术上下文：**

repo-research-analyst 的输出包含结构化的技术栈和基础设施摘要。利用它做出更精确的外部研究决策：

- 如果检测到特定框架和版本（例如 React 18、Next.js 14、Go 1.22），将这些确切标识符传递给 framework-docs-researcher，使其获取版本特定的文档
- 如果功能涉及仓库中已充分建立的技术层（例如，在计划新的后台任务时已有 Sidekiq 任务），倾向于跳过外部研究——本地模式可能已足够
- 如果功能涉及仓库中缺失或薄弱的技术层（例如，在计划新的 gRPC 服务时没有现有的 proto 文件），倾向于外部研究——没有可遵循的本地模式
- 如果检测到部署基础设施（Docker、K8s、Serverless），在传递给下游代理的计划上下文中注明，以便它们考虑部署约束
- 如果检测到 monorepo 并已限定到特定服务，将该服务的技术上下文传递给下游研究代理——而非所有服务的聚合。如果扫描显示了工作区映射但未限定范围，在继续研究之前使用功能描述识别相关服务

**在以下情况始终倾向于外部研究：**
- 主题是高风险的：安全、支付、隐私、外部 API、迁移、合规
- 代码库缺乏相关的本地模式——此计划所需模式的直接示例少于 3 个
- 本地模式存在于相邻领域但不是确切的——例如，代码库有 HTTP 客户端但没有 webhook 接收器，或有后台任务但没有事件驱动的发布/订阅。相邻模式表明团队已熟悉该技术层，但可能不了解领域特定的陷阱。当此信号存在时，围绕领域差距具体构架外部研究查询，而非通用技术
- 用户正在探索不熟悉的领域
- 技术扫描发现相关层在代码库中缺失或薄弱

**在以下情况跳过外部研究：**
- 代码库已展示出强大的本地模式——多个直接示例（非相邻领域），最近被修改，遵循当前约定
- 用户已经知道预期的形状
- 额外的外部上下文几乎不增加实际价值
- 技术扫描发现相关层已充分建立，有现有示例可遵循

在继续之前简要宣布决策。示例：
- "你的代码库对此有成熟的模式。不进行外部研究继续。"
- "这涉及支付处理，我先研究当前的最佳实践。"

#### 1.3 外部研究（条件性）

如果步骤 1.2 表明外部研究有用，并行运行这些代理：

- Task research:best-practices-researcher(计划上下文摘要)
- Task research:framework-docs-researcher(计划上下文摘要)

#### 1.4 整合研究

总结：
- 相关的代码库模式和文件路径
- 相关的制度经验
- 外部参考资料和最佳实践（如已收集）
- 相关的 Issue、PR 或先前的实践
- 任何应实质性影响计划的约束

#### 1.4b 当研究发现外部契约面时重新分类深度

如果当前分类为**轻量级**且 Phase 1 研究发现工作触及以下任何外部契约面，重新分类为**标准**：

- 被外部系统、CI 或其他仓库消费的环境变量
- 导出的公共 API、CLI 标志或命令行接口契约
- CI/CD 配置文件（`.github/workflows/`、`Dockerfile`、部署脚本）
- 被下游消费者导入的共享类型或接口
- 被外部 URL 引用或从其他系统链接的文档

这确保流程分析（Phase 1.5）运行，信心检查（Phase 5.3）应用关键部分加分。简要宣布重新分类："重新分类为标准——此更改触及[环境变量 / 导出 API / CI 配置]，存在外部消费者。"

#### 1.5 流程和边界情况分析（条件性）

对于**标准**或**深度**计划，或当用户流程完整性仍不清晰时，运行：

- Task workflow:spec-flow-analyzer(计划上下文摘要, 研究发现)

使用输出来：
- 识别缺失的边界情况、状态转换或交接空白
- 加强需求追溯或验证策略
- 仅添加实质性改善计划的流程细节

### Phase 2：解决计划问题

从以下来源构建计划问题列表：
- 源文档中的推迟问题
- 仓库或外部研究中发现的空白
- 产生有用计划所需的技术决策

对于每个问题，决定它应该是：
- **在计划期间解决** - 答案可从仓库上下文、文档或用户选择中获知
- **推迟到实现** - 答案依赖于代码更改、运行时行为或执行时发现

仅当答案实质性影响架构、范围、排序或风险且无法负责任地推断时才询问用户。使用平台的阻塞式提问工具（可用时，见交互方式）。

**不要**在此阶段运行测试、构建应用或探测运行时行为。目标是强计划，而非部分执行。

### Phase 3：结构化计划

#### 3.1 标题和文件命名

- 使用常规格式起草清晰、可搜索的标题，例如 `feat: 添加用户认证` 或 `fix: 防止结账重复提交`
- 确定计划类型：`feat`、`fix` 或 `refactor`
- 按照仓库约定构建文件名：`docs/ae/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md`
  - 如果 `docs/ae/plans/` 不存在则创建它
  - 检查今天的日期已有文件以确定下一个序号（零填充到 3 位，从 001 开始）
  - 保持描述名称简洁（3-5 个词）并使用 kebab-case
  - 示例：`2026-01-15-001-feat-user-authentication-flow-plan.md`、`2026-02-03-002-fix-checkout-race-condition-plan.md`
  - 避免：缺失序号、模糊名称如 "new-feature"、无效字符（冒号、空格）

#### 3.2 利益相关者和影响意识

对于**标准**或**深度**计划，简要考虑谁受此更改影响——最终用户、开发者、运维、其他团队——以及这应如何影响计划。对于跨切面工作，在系统范围影响部分注明受影响方。

#### 3.3 将工作拆分为实现单元

将工作拆分为逻辑实现单元。每个单元应代表一个有意义的更改，实现者通常可以作为原子提交落地。

好的单元是：
- 聚焦于一个组件、行为或集成接缝
- 通常涉及一小簇相关文件
- 按依赖排序
- 足够具体以执行而无需预先编写代码
- 使用复选框语法标记以进行进度跟踪

避免：
- 2-5 分钟的微步骤
- 跨多个不相关关注点的单元
- 过于模糊以至于实现者仍需发明计划的单元

#### 3.4 高层技术设计（可选）

在详细说明实现单元之前，决定概览是否有助于审阅者验证预期方法。此部分传达解决方案的*形状*——各部分如何组合——而不规定实现。

**何时包含：**

| 工作涉及... | 最佳概览形式 |
|---|---|
| DSL 或 API 面设计 | 伪代码语法或契约草图 |
| 多组件集成 | Mermaid 序列图或组件图 |
| 数据管道或转换 | 数据流草图 |
| 状态密集的生命周期 | 状态图 |
| 复杂分支逻辑 | 流程图 |
| 模式/标志组合或多输入行为 | 决策矩阵（输入 -> 结果） |
| 单组件但形状不明显 | 伪代码草图 |

**何时跳过：**
- 模式明确的工作，散文和文件路径已讲述完整故事
- 直截了当的 CRUD 或遵循约定的更改
- 方法显而易见的轻量级计划

选择适合工作的媒介。当图表比伪代码更好地传达时不要默认使用伪代码，反之亦然。

用以下内容标注每个草图：*"此内容说明预期方法，是审阅用的方向指引，而非实现规范。执行代理应将其视为上下文，而非要复现的代码。"*

保持草图简洁——足够验证方向，不足以复制粘贴到生产代码中。

#### 3.4b 输出结构（可选）

对于创建新目录结构的绿地计划（新插件、服务、包或模块），包含 `## 输出结构` 部分，用文件树展示预期布局。这让审阅者在深入每个单元的细节之前看到整体形状。

**何时包含：**
- 计划在新目录层次中创建 3+ 个新文件
- 目录布局本身是一个有意义的设计决策

**何时跳过：**
- 计划仅修改现有文件
- 计划在现有目录中创建 1-2 个文件——每个单元的文件列表已足够

树是范围声明，展示预期的输出形状。它不是约束——如果实现揭示了更好的布局，实现者可以调整结构。每个单元的 `**文件：**` 部分仍是每个单元创建或修改内容的权威来源。

#### 3.5 定义每个实现单元

对于每个单元，包含：
- **目标** - 此单元实现什么
- **需求** - 它推进哪些需求或成功标准
- **依赖** - 什么必须先存在
- **文件** - 要创建、修改或测试的仓库相对文件路径（绝不使用绝对路径）
- **方法** - 关键决策、数据流、组件边界或集成说明
- **执行说明** - 可选，仅当单元受益于非默认执行姿态如测试优先或特征化测试优先时
- **技术设计** - 可选的伪代码或图表，当单元的方法不明显且仅用散文会模糊时。明确标注为方向性指导，而非实现规范
- **需遵循的模式** - 要镜像的现有代码或约定
- **测试场景** - 枚举实现者应编写的具体测试用例，根据单元的复杂度和风险调整大小。考虑下面的每个类别，并包含适用于此单元的每个类别的场景。简单的配置更改可能需要一个场景；支付流程可能需要十几个。关键在于具体程度——每个场景应命名输入、操作和预期结果，使实现者不必发明覆盖范围。对于无行为更改的单元（纯配置、脚手架、样式），使用 `测试预期：无 -- [原因]` 而非留空。
  - **正常路径行为** - 具有预期输入和输出的核心功能
  - **边界情况**（当单元有有意义的边界时）- 边界值、空输入、nil/null 状态、并发访问
  - **错误和失败路径**（当单元有失败模式时）- 无效输入、下游服务失败、超时行为、权限拒绝
  - **集成场景**（当单元跨层时）- Mock 单独无法证明的行为，例如，"创建 X 触发回调 Y，回调 Y 持久化 Z"。对于任何涉及回调、中间件或多层交互的单元都应包含
- **验证** - 实现者如何知道单元已完成，以结果而非 Shell 命令脚本表示

每个包含功能的单元都应在 `**文件：**` 中包含测试文件路径。

谨慎使用 `执行说明`。好的用法包括：
- `执行说明：先为请求/响应契约编写一个失败的集成测试。`
- `执行说明：在修改此遗留解析器之前添加特征化覆盖。`
- `执行说明：以测试优先的方式实现新的领域行为。`

不要将单元扩展为字面的 `RED/GREEN/REFACTOR` 子步骤。

#### 3.6 保持计划时和实现时的未知事项分离

如果某事重要但尚不可知，将其显式记录在推迟的实现说明中，而非假装在计划中解决了它。

示例：
- 确切的方法或辅助名称
- 接触真实代码后的最终 SQL 或查询细节
- 依赖于看到实际测试失败的运行时行为
- 一旦实现开始可能变得不必要的重构

### Phase 4：编写计划

在所有深度中使用一种计划理念。更改细节量，而非计划与执行之间的边界。

#### 4.1 计划深度指南

**轻量级**
- 保持计划紧凑
- 通常 2-4 个实现单元
- 省略增加价值不大的可选部分

**标准**
- 使用完整的核心模板，省略对此特定工作无价值的可选部分（包括高层技术设计）
- 通常 3-6 个实现单元
- 在相关时包含风险、推迟问题和系统范围影响

**深度**
- 使用完整的核心模板加有必要的可选分析部分
- 通常 4-8 个实现单元
- 当这提高清晰度时将单元分组为阶段
- 在有必要时包含备选方案、文档影响和更深的风险处理

#### 4.1b 可选深度计划扩展

对于足够大、有风险或跨切面的工作，添加真正有帮助的部分：
- **考虑过的备选方案**
- **成功指标**
- **依赖 / 前提条件**
- **风险分析与缓解**
- **分阶段交付**
- **文档计划**
- **运维 / 推出说明**
- **未来考虑** 仅当它们实质性影响当前设计时

不要将这些作为样板添加。仅在它们提高执行质量或利益相关者一致性时包含。

#### 4.3 计划规则

- **所有文件路径必须是仓库相对路径** — 永远不要使用绝对路径如 `/Users/name/Code/project/src/file.ts`。使用 `src/file.ts` 代替。绝对路径会导致计划无法跨机器、工作树和团队成员移植。当计划针对不同于文档所在仓库的其他仓库时，在计划顶部声明一次目标仓库（例如，`**目标仓库：** my-other-project`）并在全文使用仓库相对路径
- 优先使用路径加类/组件/模式引用而非脆弱的行号
- 保持实现单元可用 `- [ ]` 语法检查以进行进度跟踪
- 不包含实现代码——无 import、确切方法签名或框架特定语法
- 伪代码草图和 DSL 语法在高层技术设计部分和每个单元的技术设计字段中是允许的，当它们传达设计方向时。明确将它们标注为方向性指导，而非实现规范
- 鼓励使用 Mermaid 图，当它们澄清仅用散文难以理解的关系或流程时——ERD 用于数据模型更改、序列图用于多服务交互、状态图用于生命周期转换、流程图用于复杂分支逻辑
- 不包含 git 命令、提交消息或确切的测试命令配方
- 不将实现单元扩展为微步骤的 `RED/GREEN/REFACTOR` 指令
- 不假装执行时的问题已解决以使计划看起来完整

#### 4.4 计划文档中的视觉传达

当计划包含 4+ 个具有非线性依赖的实现单元、系统范围影响中有 3+ 个交互面、概览或问题框架中有 3+ 个行为模式/变体，或关键技术决策中有 3+ 个交互决策或备选方案中有 3+ 个备选方案时，读取 `references/visual-communication.md` 获取图表和表格指导。这涵盖计划结构的视觉辅助（依赖图、交互图、比较表）——而非解决方案设计图表，后者由 Section 3.4 覆盖。

### Phase 5：最终审查、写入文件和交接

#### 5.1 写入前审查

在最终确定之前，检查：
- 计划没有发明应在 `ae:brainstorm` 中定义的产品行为
- 如果没有源文档，有边界的计划引导建立了足够的产品清晰度以负责任地计划
- 每个主要决策都有源文档或研究作为依据
- 每个实现单元都是具体的、按依赖排序的、可实现的
- 如果测试优先或特征化测试优先的姿态是明确或强烈暗示的，相关单元以轻量的 `执行说明` 携带它
- 每个包含功能的单元从每个适用类别（正常路径、边界情况、错误路径、集成）都有测试场景——根据单元的复杂度调整大小，而非填充或削减
- 测试场景命名具体的输入、操作和预期结果而不变成测试代码
- 包含功能的单元的测试场景为空或缺失被标记为不完整——包含功能的单元必须有实际测试场景，而非仅一个注释。`测试预期：无 -- [原因]` 注释仅对非功能单元（纯配置、脚手架、样式）有效
- 推迟项是显式的而非伪装的确定性
- 如果包含高层技术设计部分，它使用了适合工作的媒介，携带了非规定性标注，且不包含实现代码（无 import、确切签名或框架特定语法）
- 每个单元的技术设计字段（如存在）是简洁和方向性的而非可复制粘贴的
- 如果计划创建新目录结构，输出结构树是否有助于审阅者看到整体形状？
- 如果范围边界列出了属于单独 PR 或任务的计划工作，它们是否在 `### 推迟到单独任务` 下而非混入真正的非目标中？
- 视觉辅助（依赖图、交互图、比较表）是否有助于读者比仅扫描散文更快地掌握计划结构？

如果计划来源于需求文档，重新阅读该文档并验证：
- 所选方法仍然匹配产品意图
- 范围边界和成功标准被保留
- 阻塞问题要么已解决、明确假设，要么已退回 `ae:brainstorm`
- 源文档的每个部分都在计划中得到处理——扫描每个部分以确认没有内容被静默遗漏

#### 5.2 写入计划文件

**必需：在呈现任何选项之前将计划文件写入磁盘。**

使用 Write 工具将完整计划保存到：

```text
docs/ae/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md
```

#### 5.3 信心检查和深化

写入计划文件后，自动评估计划是否需要加强。

**两种深化模式：**

- **自动模式**（计划生成期间的默认模式）：无需请求用户批准即可运行。用户看到正在加强的内容但无需做出决定。子代理的发现直接综合到计划中。
- **交互模式**（由 Phase 0.1 中的重新深化快速路径激活）：用户明确要求深化已有计划。子代理的发现逐个呈现供审查后再整合。用户可以接受、拒绝或讨论每个代理的发现。仅接受的发现被综合到计划中。

交互模式存在是因为按需深化是不同的用户姿态——用户已经有了一个他们投入的计划并希望精确地控制什么被更改。这适用于计划是由本技能生成的、手写的，或由其他工具产生的情况。

`ae:document-review` 和此信心检查是不同的：
- 当文档需要清晰度、简化、完整性或范围控制时使用 `ae:document-review` 技能
- 此信心检查在计划结构良好但仍需更强的支撑时加强理由、排序、风险处理和系统范围思考

**管道模式：** 此阶段在管道/disable-model-invocation 上下文中始终以自动模式运行。无需用户交互。

##### 5.3.1 分类计划深度和主题风险

从文档确定计划深度：
- **轻量级** - 小型、有边界、低模糊性，通常 2-4 个实现单元
- **标准** - 中等复杂度，一些技术决策，通常 3-6 个单元
- **深度** - 跨切面、高风险或战略性重要工作，通常 4-8 个单元或分阶段交付

构建风险概况。将这些视为高风险信号：
- 认证、授权或安全敏感行为
- 支付、账单或金融流程
- 数据迁移、回填或持久数据更改
- 外部 API 或第三方集成
- 隐私、合规或用户数据处理
- 跨接口一致性或多面行为
- 重大推出、监控或运维关注

##### 5.3.2 门控：决定是否深化

- **轻量级**计划通常不需要深化，除非它们是高风险的
- **标准**计划通常在一个或多个重要部分仍然薄弱时受益
- **深度**或高风险计划通常受益于有针对性的第二轮
- **薄弱本地支撑覆盖：** 如果 Phase 1.2 因本地模式薄弱（直接示例少于 3 个或相邻领域匹配）而触发了外部研究，始终继续进行评分，无论计划看起来有多扎实。当计划建立在不熟悉的领域上时，关于系统行为的主张更可能是假设而非已验证的事实。评分轮次成本很低——如果计划确实稳固，评分什么也不会发现并快速退出

如果计划已经看起来足够扎实且薄弱支撑覆盖不适用，报告"信心检查通过——没有部分需要加强"并跳到 Phase 5.3.8（文档审查）。无论深化是否需要，文档审查始终运行——这两个工具捕获不同类别的问题。

##### 5.3.3–5.3.7 深化执行

当深化有必要时，读取 `references/deepening-workflow.md` 获取信心评分检查清单、部分到代理分发映射、执行模式选择、研究执行、交互式发现审查和计划综合说明。执行该文件中的步骤 5.3.3 到 5.3.7，然后回到这里的 5.3.8。

##### 5.3.8–5.4 文档审查、最终检查和生成后选项

到达此阶段时，读取 `references/plan-handoff.md` 获取文档审查说明（5.3.8）、最终检查和清理（5.3.9）、生成后选项菜单（5.4）和 Issue 创建。不要更早加载此文件。文档审查是强制性的——即使信心检查已经运行也不要跳过它。

永远不要写代码！研究、决策，然后编写计划。
