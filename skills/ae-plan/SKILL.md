# 创建技术计划

**注意：当前年份是 2026 年。** 在计划日期和搜索最新文档时使用此年份。

`ae:brainstorm` 定义**做什么**。`ae:plan` 定义**怎么做**。`ae:work` 执行计划。

**当被直接调用时，始终进行计划。** 永远不要将直接调用分类为"不是计划任务"而放弃。如果输入不清晰，提出澄清问题——但始终留在计划工作流中。

本工作流产生一份持久的实现计划。它**不会**实现代码、运行测试，或从执行时结果中学习。

**重要：计划文档中的所有文件引用必须使用仓库相对路径，永远不要使用绝对路径。**

## 核心原则

1. **以需求为真相来源** - 如果有需求文档，基于它构建
2. **决策，不是代码** - 捕获方法、边界、风险和测试场景，不预先编写实现
3. **先研究再结构化** - 在最终确定计划之前探索代码库和外部指导
4. **合适的产物规模** - 小工作获得紧凑计划
5. **将计划与执行发现分离** - 在此解决计划时的问题，推迟执行时的未知事项
6. **保持计划可移植** - 作为活文档使用，不嵌入工具特定指令

## 交互方式

当可用时使用平台的 `question` 工具。每次只问一个问题。

## 工作流

### Phase 0：恢复、来源和范围

#### 0.1 恢复已有计划工作

如果用户引用了现有计划文件，或在 `docs/ae/plans/` 中有明显的近期匹配计划：
- 读取它
- 确认是就地更新还是创建新计划

**深化意图：** "深化"一词是深化的触发器。一旦计划被识别且看起来完整（所有主要章节存在、实现单元已定义、`status: active`）：
- 如果计划缺少 YAML frontmatter，路由到 `references/universal-planning.md`
- 否则短路到 Phase 5.3（信心检查）的交互模式

#### 0.1b 分类任务领域

如果任务涉及软件，继续 Phase 0.2。如果是非软件的多步骤目标，读取 `references/universal-planning.md`。

#### 0.2 查找上游需求文档

搜索 `docs/ae/brainstorms/` 中匹配 `*-requirements.md` 的文件。如果多个匹配，询问使用哪一个。

#### 0.3 将源文档作为主要输入

如果存在需求文档：阅读它，宣布作为源文档，携带所有内容（问题框架、需求、成功标准、范围边界、决策、待定问题）。不要静默省略源内容。

#### 0.4 计划引导（无需求文档或输入不清晰）

如果不存在需求文档或输入需要更多结构：
- 评估是否足够清晰以直接计划
- 如果模糊主要是产品框架，推荐 `ae:brainstorm` 但同时提供继续计划的选择
- 如果用户继续，运行计划引导建立问题框架、范围、成功标准

#### 0.5 在计划之前分类待解决问题

如果源文档有阻塞问题：仅技术/架构问题重新分类为计划拥有的工作。产品阻塞项必须解决。

#### 0.6 评估计划深度

- **轻量级** - 小型、边界明确
- **标准** - 正常功能，有一些技术决策
- **深度** - 跨切面、高风险或高模糊性

### Phase 1：收集上下文

#### 1.1 本地研究（始终运行）

并行运行：
- Task research:repo-research-analyst(范围: 技术、架构、模式)
- Task research:learnings-researcher(计划上下文摘要)

#### 1.1b 检测执行姿态信号

寻找 TDD、测试优先或特征化测试优先的信号。当信号明确时，在相关实现单元中静默携带。

#### 1.2 决定外部研究

利用 repo-research-analyst 的技术上下文做精确决策。高风险主题或薄弱本地模式时倾向于外部研究。

#### 1.3 外部研究（条件性）

如果需要，并行运行 best-practices-researcher 和 framework-docs-researcher。

#### 1.4 整合研究

总结相关代码库模式、制度经验、外部参考资料。

#### 1.4b 重新分类深度

如果轻量级计划触及外部契约面（环境变量、公共 API、CI 配置），重新分类为标准。

#### 1.5 流程和边界情况分析（条件性）

对于标准或深度计划，运行 spec-flow-analyzer。

### Phase 2：解决计划问题

从源文档、研究和仓库中构建问题列表。仅当答案实质性影响架构、范围或风险时才询问用户。**不要**在此运行测试或构建应用。

### Phase 3：结构化计划

#### 3.1 标题和文件命名

文件名：`docs/ae/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md`

#### 3.2 利益相关者和影响意识

标准或深度计划：考虑谁受此更改影响。

#### 3.3 将工作拆分为实现单元

每个单元：聚焦于一个组件或行为，按依赖排序，使用复选框语法。

#### 3.4 高层技术设计（可选）

当工作涉及 DSL、多组件集成、复杂数据流、状态密集生命周期时包含。

#### 3.5 定义每个实现单元

每个单元包含：目标、需求、依赖、文件、方法、执行说明（可选）、需遵循的模式、测试场景（正常路径、边界情况、错误路径、集成）、验证。

#### 3.6 保持计划时和实现时的未知事项分离

显式记录推迟的实现说明。

### Phase 4：编写计划

使用计划模板（轻量/标准/深度各有指南）。省略不适用的可选部分。

所有文件路径必须是仓库相对路径。不包含实现代码。鼓励使用 Mermaid 图。

### Phase 5：最终审查、写入文件和交接

#### 5.1 写入前审查

检查：计划没有发明产品行为、每个决策有依据、实现单元具体且按依赖排序、测试场景覆盖所有适用类别。

#### 5.2 写入计划文件

**必需：** 在呈现任何选项之前将计划写入磁盘。

**管道模式：** 跳过交互式问题，自动决策。

#### 5.3 信心检查和深化

评估计划是否需要加强。两种模式：
- **自动模式**（默认）：无需用户批准直接综合
- **交互模式**（深化快速路径）：逐个呈现发现供审查

**管道模式：** 始终以自动模式运行。

##### 5.3.1-5.3.2 分类和门控

从文档确定深度和风险概况。薄弱支撑覆盖：如果 Phase 1.2 触发了外部研究，始终继续评分。

##### 5.3.3-5.3.7 深化执行

当深化有必要时，读取 `references/deepening-workflow.md`。

##### 5.3.8-5.4 文档审查和交接

读取 `references/plan-handoff.md`。文档审查是强制性的。

永远不要写代码！研究、决策，然后编写计划。
