# 计划文档中的视觉传达

Section 3.4 覆盖关于*正在计划的解决方案*的图表（伪代码、Mermaid 序列图、状态图）。现有的 Section 4.3 Mermaid 规则鼓励在技术设计和每个单元字段中使用这些解决方案设计图表。本指南覆盖不同的关注点：帮助读者*导航和理解计划文档本身*的视觉辅助——依赖图、交互图和比较表，使计划结构可扫描。

视觉辅助取决于内容模式，而非计划深度分类——关于复杂多单元工作流的轻量级计划可能需要依赖图；关于直接功能的深度计划可能不需要。

**何时包含：**

| 计划描述了... | 视觉辅助 | 放置位置 |
|---|---|---|
| 4+ 个具有非线性依赖的实现单元（并行、菱形、扇出/扇入） | Mermaid 依赖图 | 实现单元标题之前或之后 |
| 系统范围影响中命名 3+ 个交互面或跨层效果 | Mermaid 交互或组件图 | 系统范围影响部分内 |
| 问题/概览涉及 3+ 个行为模式、状态或变体 | Markdown 比较表 | 概览或问题框架内 |
| 关键技术决策有 3+ 个交互决策，或备选方案有 3+ 个备选 | Markdown 比较表 | 相关部分内 |

**何时跳过：**
- 计划在直线依赖链中有 3 个或更少单元——每个单元的依赖字段已足够
- 散文已经清晰传达了关系
- 视觉会重复高层技术设计部分已展示的内容
- 视觉描述代码级别的细节（具体方法名、SQL 列、API 字段列表）

**格式选择：**
- **Mermaid**（默认）用于依赖图和交互图——5-15 个节点，无框内注释，标准流程图形状。使用 `TB`（从上到下）方向，使图表在渲染和源码形式中都保持狭窄。源码应在 diff 视图和终端中作为回退可读。
- **ASCII/框线图** 用于需要丰富框内内容的注释流程——文件路径布局、决策逻辑分支、多列空间排列。当图表的价值来自节点内的注释时比 Mermaid 更具表现力。代码块遵循 80 列最大宽度，使用垂直堆叠。
- **Markdown 表格** 用于模式/变体比较和决策/方法比较。
- 保持图表与计划成比例。6 单元线性链获得简单的 6 节点图。带扇出和扇入的复杂依赖图可能需要 10-15 个节点——如果每个节点都有其存在的理由，那没问题。
- 放置在相关位置的内联，而非单独的部分。
- 仅限计划结构层面——单元依赖、组件交互、模式比较、影响面。而非实现架构、数据模式或代码结构（那些属于 Section 3.4）。
- 散文是权威的：当视觉辅助与其周围散文不一致时，以散文为准。

生成视觉辅助后，验证它准确表示了它所说明的计划部分——正确的依赖边、无缺失面、无合并单元。
