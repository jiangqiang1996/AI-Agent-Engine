# 计划交接

本文件包含计划写入后的说明：文档审查、生成后选项和 Issue 创建。在计划文件已写入且信心检查（5.3.1-5.3.7）完成之后加载。

## 5.3.8 文档审查

在信心检查（和任何深化）之后，在计划文件上运行 `ae:document-review` 技能。将计划路径作为参数传递。到达此步骤时它是强制性的——不要因为信心检查已经运行而跳过它。这两个工具捕获不同类别的问题。

信心检查和文档审查是互补的：
- 信心检查加强理由、排序、风险处理和支撑
- 文档审查检查一致性、可行性、范围对齐，并揭示角色特定的问题

如果文档审查返回了自动应用的发现，在呈现交接选项时简要注明。如果残留 P0/P1 发现被揭示，提及它们以便用户决定是否在继续之前处理。

当文档审查返回"审查完成"时，进入最终检查。

**管道模式：** 如果从自动化工作流（如 LFG）或任何 `disable-model-invocation` 上下文调用，以 `mode:headless` 和计划路径运行 `ae:document-review`。无头模式静默应用自动修复并返回结构化发现，无需交互提示。在将控制权返回给调用者之前处理任何 P0/P1 发现。

## 5.3.9 最终检查和清理

在进入生成后选项之前：
- 确认计划在具体方面更强了，而非仅仅更长
- 确认计划边界完好
- 确认当源文档存在时，源决策被保留

如果使用了制品支撑模式：
- 在计划安全更新后清理临时暂存目录
- 如果在当前平台上清理不实际，注明制品留在了哪里

## 5.4 生成后选项

**管道模式：** 如果从自动化工作流（如 LFG）或任何 `disable-model-invocation` 上下文调用，跳过下面的交互菜单并立即将控制权返回给调用者。计划文件已写入，信心检查已运行，文档审查已运行——调用者（例如 lfg）确定下一步。

文档审查完成后，使用平台的阻塞式提问工具呈现选项。如果没有提问工具可用，在聊天中呈现编号选项并等待用户回复后再继续。

**问题：** "计划已就绪：`docs/ae/plans/YYYY-MM-DD-NNN-<type>-<name>-plan.md`。接下来你想做什么？"

**选项：**
1. **启动 `/ae-work`**（推荐） - 在当前会话中开始实现此计划
2. **创建 Issue** - 从此计划在你配置的 Issue 跟踪器中创建跟踪 Issue（GitHub 或 Linear）
3. **暂时完成** - 暂停；计划文件已保存，可稍后恢复

**在上下文中呈现额外的文档审查，而非作为菜单固定项：** 当先前的文档审查轮次揭示了用户未处理的残留 P0/P1 发现时，在菜单旁边提及它们并以散文形式提供另一轮审查（例如，"文档审查标记了 2 个 P1 发现，你可能想处理——要我在你选择之前再运行一轮吗？"）。不要将其添加到选项列表中。

根据选择：
- **启动 `/ae-work`** -> 使用计划路径调用 `/ae-work`
- **创建 Issue** -> 遵循下面的 Issue 创建部分
- **暂时完成** -> 显示简短确认，说明计划文件已保存并结束本轮
- **如果用户要求再次文档审查**（来自残留 P0/P1 发现时的上下文提示，或自由形式请求） -> 使用计划路径加载 `ae:document-review` 技能进行另一轮，然后返回选项
- **其他** -> 接受自由文本进行修订并循环回选项

## Issue 创建

当用户选择"创建 Issue"时，检测他们的项目跟踪器：

1. 读取仓库根目录的 `AGENTS.md` 并查找 `project_tracker: github` 或 `project_tracker: linear`。
2. 如果 `project_tracker: github`：

   ```bash
   gh issue create --title "<type>: <title>" --body-file <plan_path>
   ```

3. 如果 `project_tracker: linear`：

   ```bash
   linear issue create --title "<title>" --description "$(cat <plan_path>)"
   ```

4. 如果未配置跟踪器，询问用户使用哪个跟踪器。使用平台的阻塞式提问工具。如果没有提问工具可用，在聊天中询问并等待回复。选项：`GitHub`、`Linear`、`跳过`。然后：
   - 使用所选跟踪器的上述命令继续
   - 提供通过在 `AGENTS.md` 中添加 `project_tracker: <value>` 来持久化选择，其中 `<value>` 是小写的跟踪器键（`github` 或 `linear`）——而非显示标签——以便未来的运行匹配步骤 1 中的检测器并跳过此提示
   - 如果 `跳过`，返回选项而不创建 Issue

5. 如果检测到的跟踪器的 CLI 未安装或未认证，显示清晰的错误（例如，"`gh` CLI 未找到——安装它或手动创建 Issue"）并返回选项。

创建 Issue 后：
- 显示 Issue URL
- 询问是否继续到 `/ae-work`，使用平台的阻塞式提问工具
