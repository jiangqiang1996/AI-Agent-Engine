# 通用计划工作流

本文件在 ae:plan 检测到非软件任务（Phase 0.1b）时加载。它用领域无关的计划工作流替代软件特定的阶段（0.2 到 5.1）。

## 开始之前：验证分类

SKILL.md 中的检测存根将任何不明确是软件的内容路由到这里。在继续之前验证分类是否正确：

- **这实际上是软件任务吗？** 关键区别是任务类型，而非主题领域。关于 Rust 的学习指南是非软件的（产出教育内容）。Rust 库重构是软件的（修改代码）。如果这实际上是软件，返回 SKILL.md 中的 Phase 0.2。
- **这是一个快速帮助请求，不是计划任务吗？** 错误消息、事实问题和单步任务不需要计划。直接回复并退出。示例："zsh: command not found: brew"、"法国的首都是什么"。
- **管道模式？** 如果从 LFG 或任何 `disable-model-invocation` 上下文调用：输出"这是一个非软件任务。LFG 管道要求 ae:work，它仅支持软件任务。请直接使用 `/ae-plan` 进行非软件计划。"并停止。

---

## 步骤 1：评估模糊性和研究需求

在计划之前评估两件事：

**1-3 个快速问题会实质性改善此计划吗？**

- **默认：通过步骤 1b 问 1-3 个问题**，当答案会改变计划的结构或内容时。始终包含一个最终选项如"跳过——直接用合理假设制定计划"以便用户可以随时退出。
- **完全跳过问题**，仅当请求已指定所有主要变量或任务足够简单以至于合理假设能很好地覆盖时。

**研究需求——此计划是否依赖于比训练数据变化更快的事实？**

| 研究需求 | 信号 | 操作 |
|--------------|---------|--------|
| **无** | 通用、永恒或概念性计划（学习课程方法论、项目管理方法、个人目标分解） | 跳过研究。模型知识已足够。在构建计划后提供："我基于通用知识制定的。需要我搜索[研究能改善的具体内容]吗？"——例如，有来源的食谱、当前产品推荐、专家框架。仅在用户接受时执行。 |
| **推荐** | 计划引用特定地点、场馆、日期、价格、时间表、季节可用性或当前事件——任何过时信息会破坏计划的内容（关闭的餐厅、变更的价格、取消的活动、错误的季节日期）。 | 在计划之前研究。分解为 2-5 个聚焦的研究问题并分派并行网络搜索。汇总发现后再构建计划。 |

当研究被推荐时，执行它——不要只是提供。过时的推荐（关闭的餐厅、更改主题的景点、过时的价格）比没有推荐更糟。用户调用 `/ae-plan` 是因为他们想要一个好的计划，而不是关于训练数据的免责声明。

**研究分解模式：**
1. 基于任务识别 2-5 个独立的研究问题。好的问题针对模型最不确定的事实：当前价格、营业时间、可用性、最近变化、季节特定性。
2. 分派并行网络搜索（每个问题一个）。先保持查询宽泛，然后根据发现缩小。
3. 在继续计划之前将发现汇总为简要的研究摘要。

"计划本周六在西雅图的一次约会之夜"的示例：
- "2026 年西雅图 Capitol Hill 周六晚开放的最好的餐厅"
- "西雅图 [具体日期] 的活动"
- "西雅图滨水区当前状态和营业时间"

## 步骤 1b：聚焦问答

最多问 3 个针对最可能改变计划的未知因素的问题。使用平台的提问工具（可用时）。否则，在聊天中呈现编号选项并等待用户回复。

**如何问好问题：**
- 提供有信息的选项，而非开放式的空白。不要问"你什么时候去？"，试试"工作日访问排队时间短 30-40%——你的时间灵活吗？" 问题应给用户一个参考框架，而非仅仅提取信息。
- 当几个独立选择可以在一个问题中捕获时使用多选。这紧凑且尊重用户的时间。
- 始终包含一个最终选项如 **"跳过——直接用合理假设制定计划"** 以便用户可以随时退出。

聚焦于此任务特有的、会改变计划推荐内容或结构方式的未知因素。不要问超过 3 个——之后，对剩余的内容使用假设继续。

## 步骤 2：结构化计划

遵循这些质量原则创建结构化计划。不要使用软件计划模板（实现单元、测试场景、文件路径等）。

### 格式：何时规定 vs 呈现选项

并非每个计划都应该是单一线性路径。将格式匹配到任务：

| 任务类型 | 最佳格式 | 原因 |
|-----------|------------|-----|
| **高个人偏好**（食物、娱乐、活动、礼物） | 每个类别精选选项——呈现 2-3 个选择让用户组合 | 偏好各异；单一选择可能不合适。选项尊重用户的品味。 |
| **逻辑序列**（学习计划、项目时间线、多天旅行物流） | 带有清晰排序的单一规定路径 | 排序重要；每一步的选项会造成决策瘫痪。 |
| **混合**（有固定结构但细节可变的活动） | 带有选择点标记的固定结构 | 骨架是固定的但具体供应商/场馆/活动是选项。 |

示例：约会夜晚计划应呈现 2-3 个餐厅选项、2-3 个活动选项和建议的流程——而非选一个餐厅然后围绕它构建整个晚上。学习计划应规定单一的每周进度——而非呈现 3 个不同的课程选择。

### 格式化：要点优于散文

- 可执行内容（步骤、选项、物流、预算）优先使用要点和表格
- 仅在上下文、理由或连接各点的解释时使用散文
- 计划用于扫描和执行，而非从头到尾阅读

### 质量原则

- **可执行的步骤**：每个步骤足够具体以执行而无需进一步研究
- **按依赖排序**：步骤按正确顺序排列，注明依赖
- **时间感知**：相关时包含时间、时长、截止日期或阶段
- **资源已识别**：指定需要什么——工具、材料、人员、预算、地点
- **应急感知**：对于重要决策，注明备选方案或计划变化时怎么做
- **适当详细**：将细节匹配到任务复杂度。周末旅行比 3 个月课程需要更少结构。晚餐计划应简洁，而非 200 行文档。
- **领域适当的格式**：选择适合领域的结构：
  - 旅行的行程（按天，带时间和地点）
  - 学习计划的课程大纲（主题、资源、里程碑）
  - 活动的运行手册（时间线、责任、物流）
  - 商业或运维任务的项目计划（阶段、负责人、交付物）
  - 调查的研究计划（问题、方法、来源）
  - 偏好驱动任务的选项菜单（每个类别精选选择）

## 步骤 3：保存或分享

结构化计划后，使用平台的提问工具询问用户希望如何接收。否则，在聊天中呈现编号选项。

**问题：** "计划已就绪。你希望如何接收？"

**选项：**

1. **保存到磁盘** — 将计划写为 Markdown 文件。询问保存位置：
   - `docs/ae/plans/`（仅当此目录存在时显示）
   - 当前工作目录
   - `/tmp`
   - 自定义路径
   - 使用文件名约定：`YYYY-MM-DD-<descriptive-name>-plan.md`
   - 以 `# 标题` 标题开始文档，下一行是 `Created: YYYY-MM-DD`。无 YAML frontmatter。

2. **暂时完成** — 计划以 Markdown 呈现在聊天中，不保存到磁盘。

不要提供 `/ae-work`（仅限软件）或 Issue 创建（不适用于非软件计划）。
