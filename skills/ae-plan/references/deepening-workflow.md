# 深化工作流

本文件包含信心检查执行路径（5.3.3-5.3.7）。仅在 5.3.2 的深化门控确定深化有必要时加载。

## 5.3.3 评分信心差距

使用检查清单优先、风险加权的评分轮次。

对于每个部分，计算：
- **触发计数** - 适用的检查清单问题数量
- **风险加分** - 如果主题是高风险且此部分与该风险实质性相关，加 1
- **关键部分加分** - 在 `标准` 或 `深度` 计划中，对 `关键技术决策`、`实现单元`、`系统范围影响`、`风险与依赖` 或 `未解决问题` 加 1

如果一个部分满足以下条件，将其视为候选：
- 达到 **2+ 总分**，或
- 在高风险领域中达到 **1+ 分**且该部分实质性重要

仅选择得分最高的 **2-5** 个部分。如果深化轻量级计划（高风险例外），上限为 **1-2** 个部分。

如果计划已有 `deepened:` 日期：
- 优先选择尚未被实质性加强的部分（如果它们的得分相当）
- 仅当已深化的部分得分明确高于其他选项时才重新审视

**部分检查清单：**

**需求追溯**
- 需求模糊或与实现单元脱节
- 成功标准缺失或未在下游反映
- 单元未明确推进已追溯的需求
- 源文档的需求未被明确携带

**上下文与研究 / 来源与参考**
- 相关的仓库模式被命名但从未用于决策或实现单元
- 引用的经验或参考资料未实质性影响计划
- 高风险工作缺乏适当的外部或内部支撑
- 研究是通用的而非针对此仓库或此计划

**关键技术决策**
- 决策被陈述但无理由
- 理由未解释权衡或被拒绝的备选方案
- 决策未联系回范围、需求或源上下文
- 存在明显的设计分支但计划从未说明为什么选择了一条路

**未解决问题**
- 产品阻塞项被隐藏为假设
- 计划拥有的问题被错误地推迟到实现
- 已解决的问题在仓库上下文、研究或源决策中没有明确依据
- 推迟项过于模糊，以后无法使用

**高层技术设计（存在时）**
- 草图使用了不适合工作的媒介
- 草图包含实现代码而非伪代码
- 非规定性标注缺失或薄弱
- 草图未连接到关键技术决策或实现单元

**高层技术设计（缺失时）** *（仅标准或深度计划）*
- 工作涉及 DSL 设计、API 面设计、多组件集成、复杂数据流或状态密集的生命周期
- 关键技术决策通过视觉或伪代码表示会更容易验证
- 实现单元的方法部分薄弱，更高层的技术设计会提供上下文

**实现单元**
- 依赖顺序不清晰或可能错误
- 文件路径或测试文件路径在应该显式的地方缺失
- 单元太大、太模糊或被拆分为微步骤
- 方法说明薄弱或未命名要遵循的模式
- 测试场景模糊（未命名输入和预期结果）、跳过适用类别（例如，有失败模式的单元没有错误路径，跨层的单元没有集成场景），或与单元的复杂度不成比例
- 包含功能的单元的测试场景为空或缺失（包含功能的单元需要实际测试场景；`测试预期：无` 注释仅对非功能单元有效）
- 验证结果模糊或未表达为可观察的结果

**系统范围影响**
- 受影响的接口、回调、中间件、入口点或一致性面缺失
- 失败传播探索不足
- 状态生命周期、缓存或数据完整性风险在相关处缺失
- 跨层工作的集成覆盖薄弱

**风险与依赖 / 文档 / 运维说明**
- 风险被列出但没有缓解措施
- 推出、监控、迁移或支持影响在有需要时缺失
- 外部依赖假设薄弱或未陈述
- 安全、隐私、性能或数据风险在明显适用处缺失

使用计划自身的 `上下文与研究` 和 `来源与参考` 作为证据。如果这些部分引用了从未影响决策、实现单元或验证的模式、经验或风险，将其视为信心差距。

## 5.3.4 报告和分发有针对性的研究

在分发代理之前，报告正在加强哪些部分以及原因：

```text
正在加强 [部分名称] — [每个的简要原因，例如，"决策理由薄弱"、"跨边界影响未映射"]
```

对于每个选定部分，选择最小有用的代理集。**不要**运行每个代理。每个部分最多使用 **1-3 个代理**，总共通常不超过 **8 个代理**。

在 Task 调用中使用完全限定的代理名称。

**固定的部分到代理映射：**

**需求追溯 / 未解决问题分类**
- `workflow:spec-flow-analyzer` 用于缺失的用户流程、边界情况和交接空白
- `research:repo-research-analyst`（范围：`架构、模式`）用于仓库支撑的模式、约定和实现现实检查

**上下文与研究 / 来源与参考空白**
- `research:learnings-researcher` 用于制度知识和过去已解决的问题
- `research:framework-docs-researcher` 用于官方框架或库行为
- `research:best-practices-researcher` 用于当前的外部模式和行业指导
- 仅当历史理由或先前的实践实质性缺失时添加 `research:git-history-analyzer`

**关键技术决策**
- `review:architecture-strategist` 用于设计完整性、边界和架构权衡
- 当决策需要仓库证据之外的外部支撑时添加 `research:framework-docs-researcher` 或 `research:best-practices-researcher`

**高层技术设计**
- `review:architecture-strategist` 用于验证技术设计是否准确表示预期方法并识别空白
- `research:repo-research-analyst`（范围：`架构、模式`）用于将技术设计基于现有仓库模式和约定
- 当技术设计涉及可从外部验证中受益的 DSL、API 面或模式时添加 `research:best-practices-researcher`

**实现单元 / 验证**
- `research:repo-research-analyst`（范围：`模式`）用于具体的文件目标、要遵循的模式和仓库特定的排序线索
- `review:pattern-recognition-specialist` 用于一致性、重复风险和与现有模式的对齐
- 当排序依赖于用户流程或交接完整性时添加 `workflow:spec-flow-analyzer`

**系统范围影响**
- `review:architecture-strategist` 用于跨边界影响、接口面和架构连锁影响
- 添加匹配风险的特定专家：
  - `review:performance-oracle` 用于可扩展性、延迟、吞吐量和资源风险分析
  - `review:security-sentinel` 用于认证、验证、攻击面和安全边界审查
  - `review:data-integrity-guardian` 用于迁移、持久状态安全、一致性和数据生命周期风险

**风险与依赖 / 运维说明**
- 使用匹配实际风险的专家：
  - `review:security-sentinel` 用于安全、认证、隐私和攻击风险
  - `review:data-integrity-guardian` 用于持久数据安全、约束和事务边界
  - `review:data-migration-expert` 用于迁移现实性、回填和生产数据转换风险
  - `review:deployment-verification-agent` 用于推出检查清单、回滚规划和上线验证
  - `review:performance-oracle` 用于容量、延迟和扩展关注

**代理提示形状：**

对于每个选定部分，传递：
- 上述映射中的范围前缀（当代理支持范围调用时）
- 简短的计划摘要
- 确切的部分文本
- 为什么选择该部分，包括哪些检查清单触发器被触发
- 计划深度和风险概况
- 要回答的具体问题

指示代理返回：
- 改变计划质量的发现
- 更强的理由、排序、验证、风险处理或参考
- 不返回实现代码
- 不返回 Shell 命令

## 5.3.5 选择研究执行模式

使用能工作的最轻量模式：

- **直接模式** - 默认。当选定的部分集较小且父级可以安全地内联读取代理输出时使用。
- **制品暂存模式** - 仅当选定的研究范围足够大以至于内联返回会产生不必要的上下文压力时使用。

证明制品暂存模式合理的信号：
- 可能超过 5 个代理返回有意义的发现
- 选定的部分摘录足够长以至于在多个代理输出中重复会是浪费的
- 主题是高风险且可能吸引大量带来源的分析

如果制品暂存模式不是明显有必要的，保持直接模式。

制品暂存模式使用每次运行的 OS 临时暂存目录。在分派子代理之前创建一次并捕获其**绝对路径**——将该绝对路径传递给每个子代理，使其直接写入。制品是每次运行的一次性产物，在深化结束时清理（见 5.3.6b），与仓库暂存空间约定匹配，用于一次性制品。不要将未解析的 Shell 变量字符串传递给子代理；它们需要已解析的绝对路径。

```bash
SCRATCH_DIR="$(mktemp -d -t ae-plan-deepen-XXXXXX)"
echo "$SCRATCH_DIR"
```

在本工作流的其余部分中，将回显的绝对路径称为 `<scratch-dir>`。

## 5.3.6 运行有针对性的研究

使用上面选择的执行模式并行启动选定的代理。如果当前平台不支持并行分派，改为顺序运行。分派时省略 `mode` 参数，以便使用用户配置的权限设置。

优先使用本地仓库和制度证据。仅当无法从仓库上下文或已引用来源负责任地弥补差距时使用外部研究。

如果选定的部分可以通过更仔细地阅读源文档来改善，在分派外部代理之前先做那件事。

**直接模式：** 让每个选定的代理将其发现直接返回给父级。保持返回负载聚焦：仅最强发现、重要的证据或来源、发现所暗示的具体计划改进。

**制品暂存模式：** 对于每个选定的代理，传递前面捕获的绝对 `<scratch-dir>` 路径，并指示代理在该目录中写入一个紧凑的制品文件，然后仅返回一个简短的完成摘要。每个制品应包含：目标部分、为什么被选中、3-7 个发现、有来源支撑的理由、每个发现暗示的具体计划更改。不包含实现代码，不包含 Shell 命令。

如果某个制品缺失或明显格式错误，重新运行该代理或对该部分回退到直接模式推理。

如果代理输出冲突：
- 优先使用仓库支撑和源文档支撑的证据而非通用建议
- 当冲突是关于库行为时，优先使用官方框架文档而非次要的最佳实践摘要
- 如果真正的权衡仍然存在，在计划中显式记录

## 5.3.6b 交互式发现审查（仅交互模式）

在自动模式中跳过此步骤——直接进入 5.3.7。

在交互模式中，在整合之前向用户呈现每个代理的发现。对于每个返回发现的代理：

1. **总结代理及其目标部分** — 例如，"architecture-strategist 审查了关键技术决策并发现："
2. **简洁地呈现发现** — 用要点列出关键观点，而非原始代理输出。包含足够的上下文让用户评估：代理发现了什么、支持它的证据、以及它暗示什么计划更改。
3. **询问用户**，使用平台的阻塞式提问工具（可用时，见交互方式）：
   - **接受** — 将这些发现整合到计划中
   - **拒绝** — 完全丢弃这些发现
   - **讨论** — 用户想在决定之前讨论这些发现

如果用户选择"讨论"，就发现进行简短对话，然后仅用接受/拒绝重新询问（第二次询问没有讨论选项）。用户无论如何都要做出明确选择。

当呈现来自针对同一部分的多个代理的发现时，一次呈现一个代理，以便用户可以做出独立决策。在展示之前不要合并来自不同代理的发现。

所有代理被审查后，仅将接受的发现带到 5.3.7。

如果用户没有接受任何发现，报告"未接受任何发现——计划未更改。" 然后直接进入 Phase 5.4（跳过 document-review 和综合——计划未被修改）。此仅交互模式的跳过不适用于自动模式；自动模式始终通过 5.3.7 和 5.3.8 进行。无需显式暂存清理——`$SCRATCH_DIR` 是 OS 临时目录，将由 OS 清理；保留它可以保存被拒绝的代理制品以供调试。

如果发现被接受且计划被修改，正常通过 5.3.7 和 5.3.8 进行——document-review 作为更改的质量门控。

## 5.3.7 综合并更新计划

仅加强选定的部分。保持计划连贯并保留其整体结构。

**在交互模式中：** 仅整合用户在 5.3.6b 中接受的发现。如果来自不同代理的一些发现触及同一部分，连贯地协调它们但不要重新引入被拒绝的发现。

允许的更改：
- 澄清或加强决策理由
- 加强需求追溯或源文档保真度
- 当排序薄弱时重新排序或拆分实现单元
- 添加缺失的模式引用、文件/测试路径或验证结果
- 在有理由的地方扩展系统范围影响、风险或推出处理
- 当证据支持更改时在 `在计划期间已解决` 和 `推迟到实现` 之间重新分类未解决问题
- 当工作需要且当前表示薄弱时加强、替换或添加高层技术设计部分
- 当单元的方法不明显时加强或添加每个单元的技术设计字段
- 当计划被实质性改善时在 frontmatter 中添加或更新 `deepened: YYYY-MM-DD`

**不要**：
- 添加实现代码——无 import、确切方法签名或框架特定语法。伪代码草图和 DSL 语法是允许的
- 添加 git 命令、提交编排或确切的测试命令配方
- 到处添加通用的 `研究洞察` 子部分
- 从头重写整个计划
- 在不显式提出的情况下发明新的产品需求、范围更改或成功标准

如果研究揭示了应改变行为或范围的产品层面模糊性：
- 不要在这里静默决定
- 记录在 `未解决问题` 中
- 如果差距真正是产品定义性的，推荐 `ae:brainstorm`
