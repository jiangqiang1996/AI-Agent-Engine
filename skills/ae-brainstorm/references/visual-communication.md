# 需求数据文档中的视觉沟通

视觉辅助以内容模式为条件，而非以深度分类为条件 —— 关于复杂工作流的轻量头脑风暴可能需要图表；关于简单功能的深入头脑风暴可能不需要。

**何时包含：**

| 需求描述的是... | 视觉辅助 | 放置位置 |
|---|---|---|
| 多步骤用户工作流或流程 | Mermaid 流程图或带注释的 ASCII 流程图 | 在问题框架之后，或对于大量流程（>10 节点）放在其自己的 `## 用户流程` 标题下 |
| 3+ 种行为模式、变体或状态 | Markdown 比较表 | 在需求部分内 |
| 3+ 个交互参与者（用户角色、系统组件、外部服务） | Mermaid 或 ASCII 关系图 | 在问题框架之后，或放在其自己的 `## 架构` 标题下 |
| 正在比较的多个竞争方案 | 比较表 | 在阶段 2 方案探索中 |

**何时跳过：**
- 文字已经清晰地传达了概念
- 图表只是以视觉形式重述需求，没有增加理解价值
- 视觉描述的是实现架构、数据 Schema、状态机或代码结构（那些属于 `ae:plan`）
- 头脑风暴简单且线性，没有多步骤流程、模式比较或多参与者交互

**格式选择：**
- **Mermaid**（默认）用于简单流程 —— 5-15 个节点，无框内注释，标准流程图形状。使用 `TB`（从上到下）方向，使图表在渲染和源码形式下都保持窄幅。源码应在 diff 视图和终端中作为后备可读。
- **ASCII/制图字符**用于需要丰富框内内容的注释流程 —— 每步的 CLI 命令、决策逻辑分支、文件路径布局、多列空间排列。当图表的价值来自步骤内的注释时，比 Mermaid 更具表现力。代码块遵循 80 列最大宽度，使用垂直堆叠。
- **Markdown 表格**用于模式/变体比较和方案比较。
- 保持图表与内容成比例。简单的 5 步工作流用 5-10 个节点。每个步骤都有决策分支和注释的复杂工作流可能需要 15-20 个节点 —— 如果每个节点都物有所值，那是可以的。
- 在相关位置内联放置，而不是放在单独的章节中。
- 仅概念层面 —— 用户流程、信息流、模式比较、组件职责。不是实现架构、数据 Schema 或代码结构。
- 文字是权威的：当视觉辅助与周围文字不一致时，以文字为准。

生成视觉辅助后，验证它是否准确代表了文字需求 —— 正确的序列、没有遗漏的分支、没有合并的步骤。没有代码可验证的图表比有代码支撑的图表有更高的不准确风险。
