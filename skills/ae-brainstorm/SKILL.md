# 功能或改进的头脑风暴

**注意：当前年份是 2026 年。** 在为需求数据文档标注日期时使用此年份。

头脑风暴通过协作对话回答**构建什么（WHAT）**。它在 `ae:plan` 之前进行，后者回答**如何构建（HOW）**。

此工作流的持久输出是一份**需求数据文档**。在其他工作流中，这可能被称为轻量级 PRD 或功能简报。在 AE 中，保留工作流名称 `brainstorm`，但要确保书面产物足够扎实，使规划阶段不需要再发明产品行为、范围边界或成功标准。

此技能不实现代码。它探索、澄清并记录决策，供后续规划或执行使用。

**重要：生成的文档中所有文件引用必须使用仓库相对路径（例如 `src/models/user.rb`），绝不能使用绝对路径。**

## 核心原则

1. **先评估范围** - 根据工作的大小和模糊程度匹配相应的流程规范。
2. **做思考伙伴** - 建议替代方案、质疑假设、探索假设情境，而不是仅仅提取需求。
3. **在此解决产品决策** - 面向用户的行为、范围边界和成功标准属于此工作流。详细的实现属于规划阶段。
4. **默认不将实现细节写入需求数据文档** - 除非头脑风暴本身涉及技术或架构变更，否则不要包含库、Schema、接口、文件布局或代码级设计。
5. **合理调整产物规模** - 简单的工作获得紧凑的需求数据文档或简短对齐。较大的工作获得更完整的文档。
6. **将 YAGNI 应用于维护成本而非编码工作量** - 优先选择能交付有意义价值的最简方案。

## 交互规则

1. **一次只问一个问题** - 不要在一次消息中批量提出多个不相关的问题。
2. **优先使用单选** - 在选择方向、优先级或下一步时使用单选。
3. **谨慎使用多选** - 仅用于可共存的集合（目标、约束、非目标）。
4. **使用平台的提问工具** - 优先使用 opencode 的 `question` 工具。

## 输出指引

- **保持输出简洁** - 优先使用简短的章节、简明的要点。
- **使用仓库相对路径** - 引用文件时使用相对路径，绝不使用绝对路径。

## 执行流程

### 阶段 0：恢复、评估和路由

#### 0.1 在适当时恢复已有工作

如果用户引用了现有的头脑风暴主题或文档，或者在 `docs/ae/brainstorms/` 中有明显的最近匹配的 `*-requirements.md` 文件：
- 阅读该文档
- 在恢复之前与用户确认
- 如果恢复，从其现有决策和待定问题继续，更新现有文档而非创建重复文档

#### 0.1b 分类任务领域

在进入阶段 0.2 之前，分类这是否是软件任务。

**软件**（继续到阶段 0.2）— 任务涉及代码、仓库、API、数据库，或要求构建/修改/调试/部署软件。

**非软件头脑风暴**（路由到通用头脑风暴）— 两个条件必须同时满足：软件信号均不存在，且任务描述的是用户想要在非软件领域探索的内容。

**两者都不是**（直接回复，跳过所有头脑风暴阶段）— 输入是快速帮助请求、错误消息、事实性问题。

**如果检测到非软件头脑风暴：** 阅读 `references/universal-brainstorming.md` 并使用其中的引导原则。不要遵循下面的软件头脑风暴阶段。

#### 0.2 评估是否需要头脑风暴

**需求清晰的指标：** 提供了具体的验收标准、引用了现有模式、描述了精确的预期行为、范围受限且定义明确。

**如果需求已经清晰：** 保持交互简短。确认理解并呈现简洁的下一步选项。仅当对规划或后续审查的持久交接有价值时才编写简短的需求数据文档。跳过阶段 1.1 和 1.2——直接进入阶段 1.3 或阶段 3。

#### 0.3 评估范围

使用功能描述加上轻量仓库扫描来分类工作：

- **轻量** - 小型、边界明确、低模糊性
- **标准** - 常规功能或有边界的重构，需要做一些决策
- **深入** - 跨领域的、战略性的或高度模糊的

### 阶段 1：理解想法

#### 1.1 现有上下文扫描

在实质性头脑风暴之前扫描仓库。根据范围匹配深度：

**轻量** — 搜索主题，检查是否已有类似内容，然后继续。

**标准和深入** — 两轮扫描：约束检查（`AGENTS.md`）和主题扫描（相关术语、现有产物、邻近示例）。

约束规则：
1. **声称前先验证** — 涉及可检查的基础设施时，阅读相关源文件确认实际存在的内容。未验证的断言必须标记为未经验证的假设。
2. **将设计决策推迟到规划** — 实现细节属于规划阶段，除非头脑风暴本身是关于技术/架构决策的。

#### 1.2 产品压力测试

在生成方案之前，质疑请求以发现错误的框架。根据范围匹配深度：

**轻量：** 是否解决真实问题？是否重复已有内容？是否有更好的框架？

**标准：** 这是正确的问题吗？什么成果真正重要？如果什么都不做会怎样？现在单一最高杠杆的行动是什么？

**深入** — 标准问题加上：这应创造什么持久能力？是推动前进还是局部补丁？

#### 1.3 协作对话

遵循交互规则。使用 opencode 的 `question` 工具进行提问。

- 先问用户已经在考虑什么
- 从广泛开始（问题、用户、价值），然后收窄（约束、排除、边界情况）
- 使需求足够具体，让规划不需要再发明行为
- 带来想法、替代方案和挑战

**退出条件：** 继续直到想法清晰，或用户明确表示要继续。

### 阶段 2：探索方案

如果仍然存在多个可行的方向，提出 **2-3 个具体方案**。否则直接陈述推荐方向。

至少使用一个非显而易见的角度——逆向思维、约束移除、或类比。

先展示方案，再评估。让用户在听到推荐之前看到所有选项。

对于每个方案：简要描述、优缺点、关键风险或未知因素、最适合的场景。

在展示所有方案后，陈述你的推荐并解释原因。

如果一个方案明显最优且替代方案没有意义，跳过菜单直接陈述推荐。

### 阶段 3：捕获需求

仅当对话产生了值得保留的持久决策时，才编写或更新需求数据文档。阅读 `references/requirements-capture.md` 获取文档模板、格式规则和完整性检查。

对于**轻量**头脑风暴，保持文档紧凑。当用户只需要简短对齐且不需要保留持久决策时，跳过文档创建。

### 阶段 3.5：文档审查

当创建或更新了需求数据文档时，在呈现交接选项之前对其运行 `ae:document-review` 技能。

如果文档审查返回了自动应用的发现，在呈现交接选项时简要提及。如果暴露了剩余的 P0/P1 发现，提及它们以便用户决定是否在继续之前解决。

当文档审查返回"审查完成"时，进入阶段 4。

### 阶段 4：交接

呈现下一步选项并执行用户的选择。阅读 `references/handoff.md` 获取选项逻辑、调度指令和结束摘要格式。
