# AE 规范保存

项目规范管理助手，严格按以下流程执行。

**重要约束**：
1. 规范只能保存为**项目级规范**，写入 `.opencode/rules/` 目录。禁止保存到全局目录
2. 项目规范的范围覆盖 `opencode.json` 中 `instructions` 数组引用的所有文件（包括 glob 匹配、明确路径、`@` 前缀引用的文件）

## 第一步：识别候选规范

回顾当前会话，识别值得持久化的通用约定。关注以下两类信号：

1. **用户的纠正与强调** — 用户明确纠正了做法、强调的偏好、要求遵循的约束
2. **反复出现的模式** — 本次会话中多次出现且被用户默认认可的约定

涵盖领域：
- 编码约定（命名、风格、模式）
- 架构决策（模块划分、依赖关系、分层策略）
- 工作流规范（测试策略、Git 工作流、部署流程）
- 工具配置（构建工具、lint 规则、CI/CD）
- 业务规则（领域约束、业务逻辑约定）
- API 设计（接口风格、错误处理、版本策略）

### 通用性过滤

每条内容必须**同时满足**以下条件：

- 适用于**整个项目**而非某次特定任务
- **未来遇到同类场景时都应遵循**
- 是**通用的、可复用的**约定
- 表述为**规则或原则**

**必须排除**：仅适用于特定函数/文件的做法、具体的配置值或硬编码路径、一次性调试步骤、框架默认行为、不具备可复用性的实现细节。

如果过滤后没有值得保存的内容，告知用户并停止。

## 第二步：会话历史交叉验证

此步骤内联执行会话历史验证逻辑，验证候选规范的历史维度。解决三个问题：

1. **反复性** — 该约定是否在多个历史会话/产物中反复出现？（反复出现 = 更值得固化）
2. **冲突检测** — 已有规范文件中是否存在相关内容？（互补 = 合并，冲突 = 需谨慎）
3. **决策背景** — 历史上是否有相关讨论？（已被否决的约定 = 必须警告用户）

### 验证执行方式

使用 Task 工具以子代理方式验证，将所有候选规范的关键词一次性传入：

```
请验证以下候选规范的历史背景：

1. {候选规范 1 的核心关键词}
2. {候选规范 2 的核心关键词}
...

搜索范围：
- 项目规范文件（.opencode/rules/ 目录下所有 .md 文件）
- opencode.json 中 instructions 引用的所有文件
- 项目产物（docs/ 目录下的文件）
- Git 历史（近期提交信息）

对每条候选规范，返回：反复性判定、已有规范状态、决策上下文。
```

### 会话历史内省

同时回顾当前会话的完整对话历史，提取以下信息辅助验证：

- **决策上下文** — 当前会话中关于候选规范的讨论背景、做出决策的原因
- **反复出现验证** — 相关话题在本次会话中出现的次数和一致性
- **用户明确态度** — 用户是否对相关约定表达过明确赞同或否定

### 已有规范冲突检查

读取 `opencode.json` 的 `instructions` 引用的所有文件，检查与候选规范的重叠：

- **没有相关文件**：标记为"新建"
- **找到互补内容**：标注为"已有互补"
- **找到冲突内容**：逐条列出冲突点

### 信号标注

| 信号 | 含义 | 处理方式 |
|------|------|---------|
| 反复出现 | 历史中多次出现相关约定 | **强烈推荐**保存 |
| 首次出现 | 历史中未找到相关记录 | 推荐，标注为"新约定" |
| 曾被否决 | 历史中有类似讨论但被明确否决 | **必须警告用户**，展示否决理由 |
| 已有互补 | 已有相关规范且内容互补 | 推荐合并到已有文件 |
| 已有冲突 | 已有规范包含矛盾约定 | 需用户明确选择覆盖或保留 |

## 第三步：展示并确认

向用户展示带有历史信号的候选规范列表：

```
发现以下可能值得保存的项目规范（共 N 条）

━━━ 规范 1 ━━━
类型：{coding/architecture/workflow/tools/domain/core}
历史信号：{信号}
内容：
{规范完整内容}

━━━ 规范 N ━━━
...

请选择：
1. 输入要保存的编号（如 "1,3"）
2. 输入 "全部" 保存所有规范
3. 输入 "取消" 放弃保存
4. 对某条规范提出修改意见（如 "2: 把 XX 改为 YY"）
```

对 `曾被否决` 的规范，必须额外展示历史否决理由。对 `已有冲突` 的规范，展示冲突点供用户裁决。

## 第四步：前置检查

读取 `opencode.json`，检查 `instructions` 数组中是否包含 `.opencode/rules/**/*.md`：

- **已存在**：继续
- **不存在**：提示用户确认是否自动添加；拒绝则停止并提示手动添加

## 第五步：写入文件

### 5.1 确定规范类型

从以下类型中选择最匹配的子目录：`core`、`coding`、`architecture`、`workflow`、`tools`、`domain`

### 5.2 冲突处理

复用第二步已收集的已有规范信息。对未覆盖的文件，补充搜索 `opencode.json` 的 `instructions` 引用的全部文件。

### 5.3 写入

保存路径：`.opencode/rules/{规范类型}/{文件名}.md`

规范文件格式：

```markdown
# {规范标题}

> 本文件由 `/ae-save-rules` 命令生成，最后更新：{当前日期}

{规范内容}
```

### 完成提示

```
规范已保存

{对每条规范列出：}
- {规范标题} → {保存路径} {新建/已合并/已更新}
```
