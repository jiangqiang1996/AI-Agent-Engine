# 第 0 步：理解与确认

仅执行一次，不计入轮次。

## 1. 解析目标

理解用户的一句话描述，明确核心目标。

如果目标为空，询问用户："你想完成什么目标？"

如果输入看起来像是文件路径（包含 `/` 或 `\` 且以 `.md` 结尾），建议用户改用 `/ae-work` 来执行计划文件。

### 技能名识别

检查输入的第一个 token 是否完整匹配已知技能名列表（ae:work, ae:review, ae:brainstorm, ae:plan, ae:plan-review, ae:document-review, ae:task-loop, ae:sql, ae:test-browser, ae:setup, ae:frontend-design, ae:save-rules, ae:handoff, ae:help, ae:update, ae:lfg）。

- 匹配时：识别为执行技能，剩余部分作为目标描述
- 不匹配时：整个输入作为目标描述，不指定执行技能

**更新此列表时同步 `ae-catalog.ts`。**

## 2. 分析执行技能交互需求（仅当指定了执行技能时）

读取目标技能的 SKILL.md（路径：`skills/<skill-name>/SKILL.md`），识别其中所有"询问用户"的节点。

### 摘要式扫描策略

对于超过 200 行的 SKILL.md，优先读取含以下关键词的段落：
- "询问""question""确认""等待用户""选择""回复"

### 交互节点分类

将识别到的交互节点分为两类：

- **可默认**（偏好类、格式类、展示顺序类）——使用合理默认值
- **不可默认**（安全类、破坏性操作类、关键路径决策类）——必须在本次确认中获得用户答案，否则列为阻断项阻止循环启动

### 降级处理

当无法完整预判时：
1. 列出已扫描到的交互节点清单
2. 标注哪些段落被跳过及原因（行数/段落数）
3. 对跳过段落给出最坏情况假设（如"若存在未识别的确认节点，将使用安全默认值跳过"）

基于具体信息由用户决定是否继续。

## 3. 探测项目工具链（仅当目标涉及代码修改时）

检查项目根目录的标志文件：

| 标志文件 | 构建命令 | 测试命令 |
|---|---|---|
| `package.json` | `npm run build` | `npm test` |
| `Cargo.toml` | `cargo build` | `cargo test` |
| `pom.xml` | `mvn compile` | `mvn test` |
| `build.gradle` / `build.gradle.kts` | `./gradlew build` | `./gradlew test` |
| `go.mod` | `go build ./...` | `go test ./...` |
| `Makefile` | 检查 build target | 检查 test target |
| 无已知标志文件 | 未检测到 | 未检测到 |

记录为 `$BUILD_CMD` 和 `$TEST_CMD`。

## 4. 分析潜在隐患

识别任务中的风险点和潜在障碍：
- 可能的破坏性操作风险
- 外部依赖不可用的风险
- 权限或环境限制
- 已知的技术约束

## 5. 推导完成条件

将用户目标拆解为 3-8 个可判定"是/否"的具体完成条件。

每个完成条件附带唯一 ID（`COND-001`、`COND-002` 等）。

如果目标过于模糊无法拆解，提示用户补充信息，或建议改用 `ae:brainstorm`。

## 6. 统一确认

向用户展示完整确认清单，等待确认后才进入循环。禁止跳过确认直接开始执行。

确认清单包含：
1. 识别到的执行技能（如有）
2. 执行技能的交互节点及预填答案（如有）
3. 不可默认节点的阻断声明（如有）
4. 探测到的工具链（如涉及代码修改）
5. 推导出的完成条件（含 ID）
6. 潜在隐患和风险点
7. 预判遗漏声明（如有）

用户可以增加、删除或修改任何完成条件。
