# 第 0 步：理解与确认

仅执行一次，不计入轮次。

## 1. 解析目标

理解用户的一句话描述，明确核心目标。

如果输入看起来像是文件路径（`.md` 结尾），建议用户改用 `/ae-work`。

### 技能名识别

检查输入的第一个 token 是否完整匹配已知技能名列表。匹配时识别为执行技能，不匹配时整个输入作为目标描述。

## 2. 分析执行技能交互需求（仅当指定了执行技能时）

读取目标技能的 SKILL.md，识别所有"询问用户"的节点。

**对于超过 200 行的 SKILL.md**，优先读取含"询问""question""确认"等关键词的段落。

**交互节点分类：**
- **可默认**（偏好类、格式类）— 使用合理默认值
- **不可默认**（安全类、破坏性操作类）— 必须获得用户答案

**降级处理：** 当无法完整预判时，列出已扫描到的节点，标注跳过的段落，给出最坏情况假设。

## 3. 探测项目工具链（仅当目标涉及代码修改时）

检查项目根目录的标志文件：

| 标志文件 | 构建命令 | 测试命令 |
|---|---|---|
| `package.json` | `npm run build` | `npm test` |
| `Cargo.toml` | `cargo build` | `cargo test` |
| `go.mod` | `go build ./...` | `go test ./...` |
| 无已知标志文件 | 未检测到 | 未检测到 |

## 4. 分析潜在隐患

识别风险点和潜在障碍：破坏性操作风险、外部依赖不可用、权限限制、已知技术约束。

## 5. 推导完成条件

将用户目标拆解为 3-8 个可判定"是/否"的具体完成条件。每个附带唯一 ID（`COND-001` 等）。

如果目标过于模糊，提示用户补充或建议 `ae:brainstorm`。

## 6. 统一确认

向用户展示完整确认清单，等待确认后才进入循环。禁止跳过确认直接开始执行。

确认清单包含：执行技能、交互节点及预填答案、不可默认节点声明、工具链、完成条件、潜在隐患、预判遗漏声明。
