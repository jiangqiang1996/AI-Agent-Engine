# 迭代任务循环

循环执行任务，每轮"执行→验证→判定"，直到目标达成、用户中止或策略穷尽时退出。

## 简介

本技能接收一句话目标描述，自动推导完成条件，然后在循环中反复派发子代理执行任务并验证结果。核心定位是**目标驱动的迭代执行器**——与 `ae:work`（计划驱动的线性执行器）互补。

**适用场景**（`ae:work` 不擅长或无法覆盖的）：

- 探索性调试："修复这个 flaky test，不确定根因在哪"
- 环境搭建："配置这个项目的 CI/CD 直到所有检查通过"
- 遗留代码修复："消除这个模块的所有 TypeScript 编译错误"

**不适用场景**：

- 有明确结构化计划的实现 → 使用 `ae:work`
- 需求尚不明确、需要探索方案 → 使用 `ae:brainstorm`
- 代码变更后的质量审查 → 使用 `ae:review`

## 输入

- **目标**：`$ARGUMENTS`（一句话描述，不支持计划文件路径）

如果输入为空，询问用户："你想完成什么目标？" 然后等待回复再继续。

如果输入看起来像是文件路径（包含 `/` 或 `\` 且以 `.md` 结尾），建议用户改用 `/ae-work` 来执行计划文件，并询问是否要改为一句话描述模式。

## 流程

整个循环严格按以下线性管线推进，每轮只走一条路径：

```
第 0 步：解析 & 确认（仅一次，不计入轮次）
    ↓
第 1 步：执行（子代理）──→ 轮次计数 +1
    ↓
第 2 步：验证（子代理）
    ↓
第 3 步：判定 ——→ 不通过 → 回到第 1 步
    ↓ 通过
   输出 DONE，结束
```

**轮次计数规则**：第 0 步不计入轮次。每启动一次第 1 步子代理，轮次 +1。达到最大轮次时第 1 步不再启动。

### 第 0 步：解析与确认（仅执行一次，不计入轮次）

1. **解析目标**：理解用户的一句话描述，明确核心目标。

2. **推导完成条件**：
   - 将用户目标拆解为 3-8 个可判定"是/否"的具体完成条件。
   - 每个完成条件附带唯一 ID（`COND-001`、`COND-002` 等）。
   - 根据任务类型，参考隐含条件推导表自动追加合理的隐含条件。
   - 如果目标过于模糊无法拆解，提示用户补充信息，或建议改用 `ae:brainstorm`。

3. **向用户展示完整验证标准，等待确认后才进入第 1 轮。禁止跳过确认直接开始执行。**
   - 用户可以增加、删除或修改任何完成条件。
   - 确认后进入第 1 轮。

### 第 1 步：执行（子代理）

**必须通过 Task 工具启动 `general` 类型子代理执行。** 当前会话不直接操作文件或运行命令。

子代理 prompt 必须包含以下结构化内容：

```
## 背景
[当前已完成哪些步骤、近 3 轮失败方案摘要（如有）、本轮是第 N 轮]

## 本轮目标
[本轮要完成的具体动作]

## 完成条件
[第 0 步确认的完成条件列表，每项附带 ID]

## 输出要求
完成后必须返回以下三项：
1. 本轮动作：具体执行了什么操作
2. 变更清单：修改/新增/删除了哪些文件
3. 执行结论：成功完成本轮目标 / 部分完成（说明差距）
```

每轮必须有实质性操作。允许子代理调用其他能力完成具体任务。

**上下文窗口管理**：历史失败摘要仅保留最近 3 轮的关键信息，早期轮次的详细信息压缩为一句话总结。

**子代理输出容错**：子代理返回后，尝试提取三项结果（动作、变更、结论）。如果输出不符合推荐格式，从自然语言中提取关键信息：
- 动作：查找"修改了""创建了""执行了"等动作描述
- 变更：查找文件路径引用
- 结论：查找"成功""失败""部分完成"等状态词

如果完全无法提取有效信息，该轮标记为"执行失败"。

子代理返回后，进入第 2 步。

### 第 2 步：验证（子代理）

**必须通过 Task 工具启动 `general` 类型子代理执行验证。** 当前会话不直接运行验证命令或读取文件内容。

子代理 prompt 必须包含以下结构化内容：

```
## 验证目标
[第 0 步确认的完成条件列表，逐项列出，每项附带 ID]

## 本轮变更
[第 1 步子代理返回的变更清单]

## 验证手段参考
[根据任务类型选择验证手段，参考验证手段参考表]

## 验证要求
逐项检查每个完成条件，对每个条件给出"通过"或"不通过"判定。
不通过时必须给出具体的失败信息（错误输出、缺失内容等）。
禁止使用"看起来没问题"等主观判断。

每个判定必须附带对应的完成条件 ID。

完成后必须返回以下三项：
1. 验证结论：全部通过 / 部分不通过
2. 逐项结果：每个完成条件的 ID、通过/不通过状态及关键证据
3. 失败摘要（如有）：不通过项的具体失败信息
```

**子代理输出容错**：如果验证子代理返回的输出不符合推荐格式：
- 尝试从自然语言中提取每个完成条件的判定结果
- 如果某个条件的判定无法提取，默认标记为"不通过"，原因记录为"验证结果不明确"

**验证子代理失败处理**：如果验证子代理超时、崩溃或完全无法返回有效结果：
- 该轮标记为"验证失败"
- 不将验证失败计入"不通过"的轮次（这是子代理问题，不是任务问题）
- 连续两轮验证失败触发退出（输出异常报告）

子代理返回后，进入第 3 步。

### 第 3 步：判定

当前会话根据两个子代理的返回，对照退出条件表做判断：

| 条件 | 动作 |
|---|---|
| 所有完成条件均已满足 | 输出完成报告，标记 DONE，结束 |
| 达到最大轮次上限 | 停止循环，输出进展与未完成原因 |
| 连续两轮无进展（见下方进展判定） | 停止循环，输出瓶颈分析 |
| 连续两轮验证不通过，且不通过项 ID 集合与上轮完全相同 | 停止循环，输出瓶颈分析 |
| 连续两轮执行子代理失败 | 停止循环，输出异常报告 |
| 连续两轮验证子代理失败 | 停止循环，输出异常报告 |
| 用户回复终止语义（"停止""结束""退出"等） | 立即停止，输出当前状态摘要 |
| 以上均不满足 | 向用户报告本轮结果和差距，等待确认后进入下一轮 |

**进展判定**（每轮必须满足至少一项）：

- (a) 产生了至少一个文件变更（`git diff` 非空），或
- (b) 验证通过项数量严格大于上轮

不满足进展要求时，该轮标记为"无进展"。连续两轮无进展触发退出。

## 每轮输出格式

当前会话在第 3 步判定后，向用户输出结构化摘要：

```markdown
### 第 N 轮结果

**本轮动作**：[执行子代理返回的动作摘要]

**验证结果**：

| 条件 ID | 条件描述 | 状态 | 证据 |
|---------|---------|------|------|
| COND-001 | [...] | ✅ 通过 | [...] |
| COND-002 | [...] | ❌ 不通过 | [失败信息] |

**差距**：[未满足的完成条件列表]
```

## 退出时输出

任何退出条件下都必须输出：

```markdown
### 循环结束

**退出原因**：[完成 / 瓶颈 / 轮次上限 / 用户终止 / 子代理异常]

**状态摘要**：
- 已完成项：[条件 ID 列表]
- 未完成项：[条件 ID 列表]

**总迭代轮次**：N

**关键变更文件**：
- [文件路径]
- [文件路径]
```

## 隐含条件推导表

根据任务类型，自动追加用户未显式声明的完成条件：

| 目标描述示例 | 用户显式目标 | 自动追加的隐含条件 |
|---|---|---|
| "实现 XX 功能" | 生成功能代码 | `npm run build` 通过 |
| "修复 XX 编译错误" | 错误消失 | `npm run build` 通过 + `npm run test` 通过 |
| "重构 XX 模块" | 完成重构 | `npm run build` 通过 + `npm run test` 通过 + 无功能行为变更 |
| "新建 XX 文件" | 创建文件 | 文件存在于文件系统 + `npm run build` 通过 |
| 通用目标 | 用户显式目标 | `npm run build` 通过（如涉及代码修改） |

用户提供了验证条件时，仍然执行推导；如果推导出用户未覆盖的条件，追加到验证标准中。

## 验证手段参考表

验证子代理根据任务类型选择最直接的验证方式：

| 任务类型 | 验证手段 |
|---|---|
| TypeScript/Node.js 代码修改 | 执行 `npm run build`；涉及业务逻辑时追加 `npm run test` |
| 文件创建/修改 | 读取目标文件，逐项检查关键内容 |
| 配置修改 | 读取配置文件，验证关键配置项的值 |
| 通用目标 | 逐项比对完成条件与实际状态 |

此表作为验证子代理 prompt 的参考，不是程序化路由。

## 执行约束

- 不自行决定最大轮次超过 10，除非用户明确传入其他值。
- 破坏性操作（删除文件、清空数据库等）依赖 opencode 权限系统控制，子代理 prompt 中建议避免破坏性操作。
- 验证不通过且未达退出条件时，等待用户确认后才进入下一轮；验证全部通过时直接输出 DONE 结束，不额外等待确认。
- 历史失败摘要每轮压缩为最近 3 轮的关键信息，避免上下文膨胀。
