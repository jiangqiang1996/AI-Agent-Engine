# 前端设计

指导创建有辨识度、生产级的前端界面，避免千篇一律的 AI 美学。本技能覆盖完整生命周期：检测现有上下文、规划设计、有意图地构建、视觉验证。

## 优先级层次

本技能中的每条规则都是默认值，而非强制要求。

1. **现有设计体系 / 代码库模式** — 最高优先级，始终遵循
2. **用户的明确指令** — 覆盖技能默认值
3. **技能默认值** — 在全新项目或用户需要设计指导时应用

在已有代码库中工作时，遵循已有模式。当用户指定的方向与默认值矛盾时，遵循用户。

## 工作流

```
检测上下文 -> 规划设计 -> 构建 -> 视觉验证
```

---

## 第零层：上下文检测

在任何设计工作之前，检查代码库中已有的设计信号。这决定了本技能中有多少主观指导适用。

### 检查内容

- **设计令牌 / CSS 变量**：`--color-*`、`--spacing-*`、`--font-*` 自定义属性、主题文件
- **组件库**：shadcn/ui、Material UI、Chakra、Ant Design、Radix 或项目特有的组件目录
- **CSS 框架**：`tailwind.config.*`、`styled-components` 主题、Bootstrap 导入、命名一致的 CSS Modules
- **排版**：HTML/CSS 中的字体导入、`@font-face` 声明、Google Fonts 链接
- **调色板**：已定义的色彩梯度、品牌色文件、设计令牌导出
- **动画库**：Framer Motion、GSAP、anime.js、Motion One、Vue Transition 导入
- **间距 / 布局模式**：一致的间距比例使用、网格系统、布局组件

使用平台的文件搜索和内容搜索工具（如 Glob/Grep）扫描这些信号，不使用 Shell 命令。

### 模式分类

根据检测到的信号，选择一种模式：

- **已有体系**（跨多个类别检测到 4+ 信号）：遵循它。技能的美学建议（排版、色彩、动效）让位于已有体系，结构性指导（构图、文案、可访问性、验证）仍然适用。
- **部分体系**（1-3 个信号）：遵循已有部分；仅在未检测到约定的领域应用技能默认值。例如，Tailwind 已配置但无组件库，则遵循 Tailwind 令牌，技能指导应用于组件结构。
- **全新项目**（未检测到信号）：完整技能指导适用。
- **模糊不清**（信号矛盾或不明确）：先询问用户再继续。

### 询问用户

当上下文模糊时，使用平台的阻塞式提问工具（`question`）。如果没有提问工具可用，假定为"部分体系"模式，保守推进。

示例问题："检测到 [信号列表]。应遵循现有设计模式还是创建独特的设计？"

## 第一层：构建前规划

在编写代码前，写出三句简短声明。这些创建一致性，并给用户一个在写代码前纠正方向的机会。

1. **视觉主题** — 一句话描述氛围、材质和能量
   - 全新项目示例："干净的编辑风格，大量留白，衬线标题，柔和的大地色调" 或 "密集的数据驱动仪表盘，等宽字体点缀，深色表面层级"
   - 已有代码库：描述*已有*美学以及新工作如何延伸它

2. **内容规划** — 页面上放什么，按什么顺序
   - 着陆页：英雄区、支撑点、细节、行动号召
   - 应用：主工作区、导航、辅助上下文
   - 组件：有哪些状态，传达什么信息

3. **交互规划** — 2-3 个具体的动效创意来改变体验感
   - 不是"加动画"，而是"英雄区加载时交错淡入、段落间滚动视差、卡片悬停时放大"
   - 已有代码库中，仅描述新增交互，使用现有动效库

## 第二层：设计指导核心

这些原则适用于所有上下文类型。每条都遵循优先级层次，让位于已有设计体系和用户指令。

### 排版

- 选择有辨识度、有个性的字体。在全新项目中避免常见选择（Inter、Roboto、Arial、系统默认），除非已有代码库使用了它们。
- 最多两种字体，除非有明确理由使用更多。将展示/标题字体与正文字体搭配。
- *检测到已有字体选择时遵循现有方案。*

### 色彩与主题

- 使用 CSS 变量构建统一的配色方案。一个主色调配以鲜明的强调色，优于胆怯的、均匀分布的配色。
- 不偏向紫色配白色，不偏向深色模式。根据上下文在浅色和深色之间切换。
- 默认一个强调色，除非产品已有多种颜色的体系。
- *检测到已有色彩令牌时遵循现有方案。*

### 构图

- 从构图开始，而非从组件开始。将首个视口当作海报来设计，而非文档。
- 在添加装饰（边框、阴影、卡片）之前，先用留白、对齐、比例、裁剪和对比。
- 默认无卡片布局。卡片仅在对用户交互充当容器时允许（可点击项、可拖拽单元、可选项）。如果移除卡片样式不影响理解，就不应该是卡片。
- *所有构图规则都是默认值，用户可以覆盖。*

### 动效

- 为视觉主导型工作交付 2-3 个有意设计的动效：一个入场序列，一个滚动关联或深度效果，一个悬停/揭示过渡。
- 如果项目已有动画库则使用它。
- 未检测到已有库时，根据框架选择默认值：
  - **CSS 动画**作为通用基线
  - **Framer Motion** 用于 React 项目
  - **Vue Transition / Motion One** 用于 Vue 项目
  - **Svelte transitions** 用于 Svelte 项目
- 动效在快速录屏中应可感知，在移动端应流畅，在整个页面中应一致。纯装饰性的动效应移除。

### 可访问性

- 默认使用语义化 HTML：`nav`、`main`、`section`、`article`、`button`——而非全部用 div。
- 色彩对比度满足 WCAG AA 最低标准。
- 所有交互元素都有焦点状态。
- 可访问性与美学并不矛盾，做得好时两者兼得。

### 图像

- 需要图片时，优先选择真实或逼真的摄影作品，而非抽象渐变或假 3D 对象。
- 选择或生成具有稳定色调区域用于文字叠加的图片。
- 如果环境中有图像生成工具，用它们创建上下文相关的视觉内容，而非占位图。

## 上下文模块

选择与构建内容匹配的模块。在已有应用中工作时，无论功能是什么，默认使用模块 C。

### 模块 A：着陆页与营销（全新项目）

**默认段落顺序：**
1. 英雄区 — 品牌/产品、承诺、行动号召、一个主导视觉
2. 支撑点 — 一个具体特性、优惠或证据
3. 细节 — 氛围、工作流、产品深度或故事
4. 最终行动号召 — 转化、开始、访问或联系

**英雄区规则（默认）：**
- 一个构图，不是仪表盘。全出血图片或主导视觉平面。
- 品牌优先，标题其次，正文再次，行动号召最后。
- 保持文字列窄，锚定在图片的安静区域。
- 除非有明确理由，否则不超过 6 个段落。
- 一个 H1 标题。折叠线以上一个主要行动号召。

**文案：**
- 让标题承载含义。辅助文案通常是一个短句。
- 用产品语言写作，不是设计评论。UI 中不出现提示词语言或 AI 评论。
- 每个段落一个任务：解释、证明、深化或转化。
- 每句话都应证明其存在价值。默认更少文案，而非更多。

### 模块 B：应用与仪表盘（全新项目）

**默认模式：**
- 平静的表面层级，强烈的排版和间距，少量色彩，密集但可读的信息，最少的装饰。
- 围绕以下组织：主工作区、导航、辅助上下文/检查器、一个明确的强调色用于操作或状态。
- 仅当卡片是交互本身（可点击项、可拖拽单元、可选项）时使用卡片。如果面板能变成纯布局而不丢失含义，移除卡片样式。

**文案（功能性，非营销）：**
- 优先关注定位、状态和操作，而非承诺、氛围或品牌语调。
- 章节标题应说明区域是什么或用户能在那里做什么。好："计划状态"、"搜索指标"。差："释放你的潜能"。
- 如果一句话可以出现在首页英雄区，就重写直到它读起来像产品 UI 文案。
- 检验标准：如果操作员只扫描标题、标签和数字，能否立即理解页面？

### 模块 C：组件与功能（已有应用中的默认选择）

用于向已有应用添加内容：

- 匹配现有视觉语言。这个模块关乎融入，而非突出。
- 从周围代码继承间距比例、圆角、色彩令牌和排版。
- 关注交互质量：清晰的状态（默认、悬停、激活、禁用、加载、错误），状态间的平滑过渡，明确的操作提示。
- 一个新组件不应引入新的设计体系。如果现有应用使用 4px 圆角，不要添加 8px 圆角的组件。

## 硬性规则与反模式

### 默认反对（可覆盖）

这些是技能的主观偏好，用户可以覆盖任何一条。

- 通用 SaaS 卡片网格作为第一印象
- 紫色配白色配色方案、深色模式偏好
- 全新项目中使用过度常见的字体（Inter、Roboto、Arial、Space Grotesk、系统默认）
- 英雄区堆砌统计数据、日程、标签集群、Logo 墙
- 段落用不同措辞重复相同氛围声明
- 没有叙事目的的轮播图
- 多个互相竞争的强调色
- 装饰性渐变或抽象背景代替真实视觉内容
- 听起来像设计评论的文案（"体验无缝集成"）
- 分屏英雄区中文字位于图片的嘈杂一侧

### 始终避免（质量底线）

这些是真正的质量缺陷，没有用户会想要。

- 提示词语言或 AI 评论泄漏到 UI 中
- 对比度问题——文字在图片或背景上不可读
- 交互元素没有可见的焦点状态
- 存在适当 HTML 元素时使用语义 div 堆砌

## 检验清单

进入视觉验证前的快速自审。并非所有检查都适用于每个上下文——判断哪些相关。

- 首屏是否一目了然地展示了品牌或产品？
- 是否有一个强烈的视觉锚点？
- 仅扫描标题能否理解页面？
- 每个段落是否只有一个任务？
- 使用卡片的地方是否确实有必要？
- 动效是否改善了层级或氛围，还是只是摆设？
- 移除所有装饰性阴影后设计是否仍然出色？
- 文案听起来像产品还是像提示词？
- 新工作是否匹配现有设计体系？（模块 C）

## 视觉验证

实现后，进行视觉验证。这是合理性检查，不是像素级审查。一轮即可。有明显问题就修复，看起来不错就继续。

### 工具优先级

按顺序使用第一个可用选项：

1. **项目已有的浏览器工具** — 如果 Playwright、Puppeteer、Cypress 等已在项目依赖中，使用它。不引入新依赖仅用于验证。
2. **浏览器 MCP 工具** — 如果环境中提供了浏览器自动化工具，使用它们。
3. **agent-browser CLI** — 如果以上都不可用且已安装 agent-browser，使用它。如果未安装 agent-browser，请先使用 /ae-setup 安装。
4. **心理审查** — 如果无法访问浏览器（无头 CI、无安装权限），将检验清单作为自审应用，并注明视觉验证已跳过。

### 评估内容

- 输出是否匹配构建前规划的视觉主题？
- 是否有明显的视觉问题（布局错乱、文字不可读、图片缺失）？
- 是否符合目标上下文模块的感觉（着陆页像着陆页，仪表盘像仪表盘，组件融入环境）？

### 范围控制

一轮迭代。截图、对照检验清单评估、修复明显问题、继续。将截图包含在交付物中（PR 描述、对话输出等）。

如需多轮截图-评估-修复的迭代细化，使用 @ae/design-iterator 代理。

完整的浏览器端到端测试请使用 /ae-test-browser。

## 创造力

本技能提供结构，但目标是做出有辨识度、避免 AI 千篇一律的作品——不是公式化输出。

对于全新项目，大胆投入一个美学方向。考虑调性：极简主义、极繁主义、复古未来、有机/自然、奢华/精致、趣味、编辑风、粗野主义、装饰艺术、柔和/粉彩、工业风——或发明适合上下文的风格。这些是灵感来源，但要设计出忠于项目的方案。

问自己：什么让这个作品令人难忘？什么是用户会记住的那一点？

实现复杂度要匹配美学愿景。极繁主义设计需要精细的代码和大量动画效果。极简主义设计需要克制、精准，以及对间距、排版和细节的精心关注。优雅来自于把愿景执行好，而非来自强度。
