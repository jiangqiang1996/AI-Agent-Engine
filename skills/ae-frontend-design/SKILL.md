# 前端设计

指导创建有辨识度、生产级的前端界面，避免千篇一律的 AI 美学。覆盖完整生命周期：检测现有上下文、规划设计、有意图地构建、视觉验证。

## 优先级层次

1. **现有设计体系 / 代码库模式** — 最高优先级，始终遵循
2. **用户的明确指令** — 覆盖技能默认值
3. **技能默认值** — 在全新项目或用户需要设计指导时应用

## 工作流

```
检测上下文 -> 规划设计 -> 构建 -> 视觉验证
```

---

## 第零层：上下文检测

在任何设计工作之前，检查代码库中已有的设计信号。

### 检查内容

- **设计令牌 / CSS 变量**：`--color-*`、`--spacing-*`、`--font-*` 自定义属性、主题文件
- **组件库**：shadcn/ui、Material UI、Chakra、Ant Design、Radix 或项目特有的组件目录
- **CSS 框架**：`tailwind.config.*`、`styled-components` 主题、Bootstrap 导入、CSS Modules
- **排版**：字体导入、`@font-face` 声明、Google Fonts 链接
- **调色板**：已定义的色彩梯度、品牌色文件、设计令牌导出
- **动画库**：Framer Motion、GSAP、anime.js、Motion One、Vue Transition 导入
- **间距 / 布局模式**：一致的间距比例使用、网格系统、布局组件

使用平台的文件搜索和内容搜索工具（如 Glob/Grep）扫描这些信号，不使用 Shell 命令。

### 模式分类

- **已有体系**（4+ 信号）：遵循它。技能美学建议让位于已有体系，结构性指导仍然适用
- **部分体系**（1-3 个信号）：遵循已有部分；仅在未检测到约定的领域应用技能默认值
- **全新项目**（未检测到信号）：完整技能指导适用
- **模糊不清**（信号矛盾）：先询问用户再继续

## 第一层：构建前规划

在编写代码前，写出三句简短声明：

1. **视觉主题** — 一句话描述氛围、材质和能量
2. **内容规划** — 页面上放什么，按什么顺序
3. **交互规划** — 2-3 个具体的动效创意来改变体验感

---

## 第二层：设计指导核心

### 排版

- 选择有辨识度、有个性的字体。全新项目中避免过度常见选择（Inter、Roboto、Arial、系统默认）
- 最多两种字体，将展示/标题字体与正文字体搭配

### 色彩与主题

- 使用 CSS 变量构建统一配色方案。一个主色调配以鲜明的强调色
- 不偏向紫色配白色，不偏向深色模式

### 构图

- 从构图开始，而非从组件开始。将首个视口当作海报来设计
- 在添加装饰之前，先用留白、对齐、比例、裁剪和对比
- 默认无卡片布局。卡片仅在对用户交互充当容器时允许

### 动效

- 为视觉主导型工作交付 2-3 个有意设计的动效
- 根据框架选择默认值：CSS 动画（通用）、Framer Motion（React）、Vue Transition / Motion One（Vue）、Svelte transitions（Svelte）

### 可访问性

- 默认使用语义化 HTML
- 色彩对比度满足 WCAG AA 最低标准
- 所有交互元素都有焦点状态

---

## 上下文模块

### 模块 A：着陆页与营销（全新项目）

**默认段落顺序：** 英雄区 → 支撑点 → 细节 → 最终行动号召

### 模块 B：应用与仪表盘（全新项目）

**默认模式：** 平静的表面层级，强烈的排版和间距，少量色彩，密集但可读的信息

### 模块 C：组件与功能（已有应用中的默认选择）

- 匹配现有视觉语言，融入而非突出
- 从周围代码继承间距比例、圆角、色彩令牌和排版
- 一个新组件不应引入新的设计体系

---

## 硬性规则

### 默认反对（可覆盖）

- 通用 SaaS 卡片网格作为第一印象
- 紫色配白色配色方案、深色模式偏好
- 英雄区堆砌统计数据、标签集群、Logo 墙
- 段落用不同措辞重复相同氛围声明
- 没有叙事目的的轮播图
- 装饰性渐变代替真实视觉内容

### 始终避免（质量底线）

- 提示词语言或 AI 评论泄漏到 UI 中
- 对比度问题 — 文字在图片或背景上不可读
- 交互元素没有可见的焦点状态
- 存在适当 HTML 元素时使用语义 div 堆砌

---

## 检验清单

- 首屏是否一目了然地展示了品牌或产品？
- 是否有一个强烈的视觉锚点？
- 仅扫描标题能否理解页面？
- 使用卡片的地方是否确实有必要？
- 动效是否改善了层级或氛围？
- 移除所有装饰性阴影后设计是否仍然出色？
- 文案听起来像产品还是像提示词？

---

## 视觉验证

实现后，按工具优先级使用第一个可用选项：

1. **项目已有的浏览器工具** — Playwright、Puppeteer 等
2. **浏览器 MCP 工具**
3. **agent-browser CLI** — 未安装则先使用 /ae-setup 安装
4. **心理审查** — 无头 CI 等环境下降级

评估内容：是否匹配构建前规划的视觉主题？是否有明显视觉问题？是否符合目标上下文模块的感觉？

一轮迭代。截图、评估、修复明显问题、继续。

如需多轮迭代细化，使用 @ae/design-iterator 代理。

完整的浏览器端到端测试请使用 /ae-test-browser。
