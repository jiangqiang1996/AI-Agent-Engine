# Diff 范围规则

以下规则适用于每位审查者。它们定义了什么是"你需要审查的代码"与预存上下文的区别。

## 范围发现

按以下优先级确定要审查的 diff：

1. **用户指定范围。** 如果调用方传入了 `BASE:`、`FILES:` 或 `DIFF:` 标记，直接使用该范围。
2. **工作副本变更。** 如果存在未暂存或已暂存的变更（`git diff HEAD` 非空），审查这些变更。
3. **未推送提交与基准分支对比。** 如果工作副本是干净的，审查 `git diff $(git merge-base HEAD <base>)..HEAD`，其中 `<base>` 是默认分支（main 或 master）。

SKILL.md 中的范围步骤负责发现并传递已解析的 diff 给你。你不需要自己运行 git 命令。

## 发现分类层级

你报告的每个发现根据其与 diff 的关系分为三个层级：

### 主要（直接变更的代码）

diff 中新增或修改的行。这是你的主要关注点。对这些行报告发现时使用完全置信度。

### 次要（紧邻的周围代码）

与变更行处于同一函数、方法或代码块中的未变更代码。如果某个变更引入的 bug 只有通过阅读周围上下文才能发现，则报告它——但需注明该问题存在于新代码与现有代码的交互中。

### 预存（与本次 diff 无关）

diff 未触及且不与之交互的未变更代码中的问题。在输出中将这些标记为 `"pre_existing": true`。它们会单独报告，不计入审查结论。

**规则：** 如果你在一个不包含该周围文件的相同 diff 上也会标记同一问题，则它是预存的。如果 diff 使该问题变得*新相关*（例如，新调用者触发了现有有 bug 的函数），则它是次要的。
