# 工作执行技能

按计划高效执行工作，保持质量并完成功能交付。

## 简介

本技能接收一份工作文档（计划、规范或待办文件）或一段描述工作的提示词，并系统化地执行。核心目标是**交付完整功能**——快速理解需求、遵循已有模式、全程保持质量。

## 输入文档

<input_document> #$ARGUMENTS </input_document>

## 执行工作流

### 阶段 0：输入分流

根据 `<input_document>` 中提供的内容决定如何推进。

**计划文档**（输入为已有的计划、规范或待办文件的路径）→ 跳至阶段 1。

**裸提示词**（输入为工作描述，非文件路径）：

1. **扫描工作区域**

   - 根据提示词识别可能需要变更的文件
   - 查找这些区域的已有测试文件（搜索与实现文件有导入、引用或命名关联的 test/spec 文件）
   - 记录受影响区域的本地模式和约定

2. **评估复杂度并路由**

   | 复杂度 | 信号 | 操作 |
   |--------|------|------|
   | **简单** | 1-2 个文件，无行为变更（拼写修正、配置调整、重命名） | 进入阶段 1 步骤 2（环境准备），然后直接实现——无需任务列表和执行循环。若变更涉及承载行为的代码，应用测试发现 |
   | **小/中** | 范围明确，约 10 个文件以内 | 基于发现构建任务列表。进入阶段 1 步骤 2 |
   | **大** | 跨模块、架构决策、10+ 文件、涉及认证/支付/数据迁移 | 告知用户建议先运行 `/ae-brainstorm` 或 `/ae-plan` 以暴露边界情况和范围边界。尊重用户选择。若继续执行，构建任务列表并进入阶段 1 步骤 2 |

---

### 阶段 1：快速启动

1. **阅读计划并澄清** _(若从阶段 0 裸提示词进入则跳过)_

   - 完整阅读工作文档
   - 将计划视为决策产物，而非执行脚本
   - 若计划包含 `Implementation Units`、`Work Breakdown`、`Requirements Trace`、`Files`、`Test Scenarios` 或 `Verification` 等章节，以此作为执行的主要素材
   - 检查每个实现单元的 `Execution note`——这些承载了计划对该单元的执行姿态信号（例如测试优先或特征化优先）。创建任务时注意记录
   - 检查 `Deferred to Implementation` 或 `Implementation-Time Unknowns` 章节——这些是计划者有意留给执行时解决的问题。开始前记录，以便在方法层面提前考虑，而非执行中途才发现
   - 检查 `Scope Boundaries` 章节——这些是明确的非目标。若实现过程中开始偏向相邻工作，回看此节
   - 阅读计划中提供的引用或链接
   - 若用户在本次会话中明确要求 TDD、测试优先或特征化优先执行，即使计划无 `Execution note`，也要遵循
   - 若有不明确或模糊之处，立即提问澄清
   - 获得用户许可后继续
   - **不要跳过此步**——现在提问比构建错误的东西要好得多

2. **准备环境**

   首先，检查当前分支：

   ```bash
   current_branch=$(git branch --show-current)
   default_branch=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@')

   # 若远程 HEAD 未设置则回退
   if [ -z "$default_branch" ]; then
     default_branch=$(git rev-parse --verify origin/main >/dev/null 2>&1 && echo "main" || echo "master")
   fi
   ```

   **若已在功能分支上**（非默认分支）：

   首先，检查分支名是否**有意义**——像 `feat/crowd-sniff` 或 `fix/email-validation` 这样的名称能让后续读者了解工作内容。自动生成的 worktree 名称（如 `worktree-jolly-beaming-raven`）或其他无意义名称则不行。

   若分支名无意义或为自动生成，建议在继续前重命名：
   ```bash
   git branch -m <meaningful-name>
   ```
   从计划标题或工作描述推导新名称（如 `feat/crowd-sniff`）。将重命名作为建议选项，同时提供继续使用当前名称的选项。

   然后询问："继续在 `[current_branch]` 上工作，还是创建新分支？"
   - 若继续（无论是否重命名），进入步骤 3
   - 若创建新分支，按下方选项 A 或 B 操作

   **若在默认分支上**，选择如何继续：

   **选项 A：创建新分支**
   ```bash
   git pull origin [default_branch]
   git checkout -b feature-branch-name
   ```
   根据工作内容使用有意义的名称（如 `feat/user-authentication`、`fix/email-validation`）。

   **选项 B：使用 worktree（推荐用于并行开发）**
   ```bash
   # 使用 git worktree 在隔离的工作树中从默认分支创建新分支
   # 示例：git worktree add ../feature-dir -b feat/new-feature
   ```

   **选项 C：继续在默认分支上工作**
   - 需要用户明确确认
   - 仅在用户明确表示"是的，提交到 [default_branch]"后才继续
   - 未经明确许可，绝不直接提交到默认分支

   **建议**：在以下情况使用 worktree：
   - 需要同时开发多个功能
   - 想在实验时保持默认分支干净
   - 计划频繁切换分支

3. **创建待办列表** _(若阶段 0 已构建，或阶段 0 路由为"简单"，则跳过)_
   - 使用可用的任务跟踪工具（如 TodoWrite、任务列表）将计划拆解为可执行任务
   - 从计划的实现单元、依赖关系、文件、测试目标和验证标准推导任务
   - 将每个单元的 `Execution note` 带入对应任务（若存在）
   - 实现前阅读每个单元的 `Patterns to follow` 字段——这些指向需要模仿的具体文件或约定
   - 将每个单元的 `Verification` 字段作为该任务的首要"完成"信号
   - 不要期望计划包含实现代码、微步骤 TDD 指令或精确的 shell 命令
   - 包含任务间的依赖关系
   - 根据优先级排序
   - 包含测试和质量检查任务
   - 保持任务具体且可完成

4. **选择执行策略**

   创建任务列表后，根据计划的规模和依赖结构决定执行方式：

   | 策略 | 适用场景 |
   |------|----------|
   | **内联** | 1-2 个小任务，或需要中途与用户交互的任务。**裸提示词工作的默认策略**——裸提示词很少产生足够的结构化上下文来支持子代理派发 |
   | **串行子代理** | 3+ 个有依赖关系的任务。每个子代理获得一个聚焦于单个单元的全新上下文窗口——防止跨任务上下文退化。需要计划单元元数据（Goal、Files、Approach、Test scenarios） |
   | **并行子代理** | 3+ 个通过并行安全检查（见下文）的任务。同时派发独立单元，依赖单元在其前置完成后运行。需要计划单元元数据 |

   **并行安全检查**——选择并行派发前的必做检查：

   1. 从每个候选单元的 `Files:` 章节（Create、Modify 和 Test 路径）构建文件到单元的映射
   2. 检查交集——任何文件路径出现在 2+ 个单元中即表示重叠
   3. 若发现重叠，降级为串行子代理。记录原因（如"单元 2 和 4 共享 `config/routes.rb`——使用串行派发"）。串行子代理仍提供上下文窗口隔离，且无共享目录风险

   即使无文件重叠，并行子代理共享工作目录仍面临 git 索引争用（并发暂存/提交会损坏索引）和测试干扰（并发测试运行会拾取彼此进行中的变更）。以下并行子代理约束可缓解这些问题。

   **子代理派发**使用可用的子代理或任务派发机制。对于每个单元，给予子代理：
   - 完整的计划文件路径（用于整体上下文）
   - 该单元的 Goal、Files、Approach、Execution note、Patterns、Test scenarios 和 Verification
   - 任何与该单元相关的已解决的延迟问题
   - 指示子代理检查该单元的测试场景是否覆盖所有适用类别（正常路径、边界情况、错误路径、集成），并在编写测试前补充缺失

   **并行子代理约束**——并行派发单元时（非串行或内联）：
   - 指示每个子代理："不要暂存文件（`git add`）、创建提交或运行项目测试套件。编排器在所有并行单元完成后处理测试、暂存和提交。"
   - 这些约束防止并发子代理间的 git 索引争用和测试干扰

   **权限模式：** 派发子代理时省略 `mode` 参数，以使用用户配置的权限设置。不要传递 `mode: "auto"`——它会覆盖用户级设置如 `bypassPermissions`。

   **每个子代理完成后（串行模式）：**
   1. 审查子代理的 diff——验证变更是否符合单元范围和 `Files:` 列表
   2. 运行相关测试套件确认代码树健康
   3. 若测试失败，诊断并修复后再继续——不要在损坏的代码树上派发依赖单元
   4. 更新计划复选框和任务列表
   5. 派发下一个单元

   **并行批次中所有子代理完成后：**
   1. 等待当前并行批次中的每个子代理完成后才处理其结果
   2. 交叉检查发现的文件冲突：比较批次中所有子代理实际修改的文件（不仅是其声明的 `Files:` 列表）。子代理可能创建或修改计划时未预见的文件——这是正常的，因为计划描述的是*做什么*而非*怎么做*。只有当同一批次中 2+ 个子代理修改了同一文件时冲突才有影响。在共享工作目录中，只有最后写入者的版本会保留——另一个单元对该文件的修改会丢失。若检测到冲突：先提交所有单元的非冲突文件，然后串行重新运行受影响单元处理共享文件，使每个单元都基于已提交的工作
   3. 对于每个已完成的单元，按依赖顺序：审查 diff、运行相关测试套件、仅暂存该单元的文件，并使用从单元 Goal 派生的约定式消息提交
   4. 若提交某个单元的变更后测试失败，诊断并修复后再提交下一个单元
   5. 更新计划复选框和任务列表
   6. 派发下一批独立单元，或下一个依赖单元

### 阶段 2：执行

1. **任务执行循环**

   按优先级顺序处理每个任务：

   ```
   while (仍有任务):
     - 将任务标记为进行中
     - 阅读计划中引用的或阶段 0 中发现的文件
     - 在代码库中查找相似模式
     - 查找正在变更的实现文件的已有测试文件（测试发现——见下文）
     - 遵循已有约定进行实现
     - 添加、更新或移除测试以匹配实现变更（见下方测试发现）
     - 运行系统级测试检查（见下文）
     - 变更后运行测试
     - 评估测试覆盖：此任务是否改变了行为？若是，是否编写或更新了测试？若未添加测试，是否有合理理由（如纯配置、无行为变更）？
     - 将任务标记为已完成
     - 评估是否增量提交（见下文）
   ```

   当单元带有 `Execution note` 时，遵循其指引。对于测试优先的单元，在该单元的实现之前先编写失败的测试。对于特征化优先的单元，在变更前先捕获现有行为。对于无 `Execution note` 的单元，务实推进。

   执行姿态护栏：
   - 测试优先工作时，不要在同一步骤中编写测试和实现
   - 不要跳过验证新测试在实现修复或功能前确实失败
   - 测试优先工作时，不要过度实现超出当前行为切片
   - 对于简单重命名、纯配置和纯样式工作，跳过测试优先纪律

   **测试发现**——在实现文件变更前，查找其已有测试文件（搜索与实现文件有导入、引用或命名关联的 test/spec 文件）。当计划指定了测试场景或测试文件时，从那里开始，然后检查计划可能未枚举的额外测试覆盖。实现文件的变更应伴随相应的测试更新——新行为添加新测试、变更行为修改测试、删除行为移除或更新测试。

   **测试场景完整性**——为承载功能的单元编写测试前，检查计划的 `Test scenarios` 是否覆盖了该单元适用的所有类别。若缺少类别或场景模糊（如"验证正确"但未说明输入和预期输出），从单元自身上下文补充后再编写测试：

   | 类别 | 适用时机 | 缺失时如何推导 |
   |------|----------|----------------|
   | **正常路径** | 始终适用于承载功能的单元 | 阅读单元的 Goal 和 Approach 获取核心输入/输出对 |
   | **边界情况** | 单元有有意义的边界（输入、状态、并发） | 识别边界值、空/nil 输入和并发访问模式 |
   | **错误/失败路径** | 单元有失败模式（验证、外部调用、权限） | 枚举单元应拒绝的无效输入、应执行的权限/认证拒绝、应处理的下游失败 |
   | **集成** | 单元跨层（回调、中间件、多服务） | 识别跨层链路，编写不使用 mock 的端到端场景 |

   **系统级测试检查**——在标记任务完成前，暂停并自问：

   | 问题 | 操作 |
   |------|------|
   | **这段代码运行时会触发什么？** 回调、中间件、观察者、事件处理器——从变更处向外追踪两层。 | 阅读所触碰模型的回调、请求链中的中间件、`after_*` 钩子的实际代码（不是文档）。 |
   | **我的测试是否覆盖了真实链路？** 若每个依赖都被 mock，测试只证明逻辑在*隔离环境*中有效——对交互无任何说明。 | 至少编写一个集成测试，使用真实对象穿过完整的回调/中间件链。交互层不使用 mock。 |
   | **失败是否会留下孤立状态？** 若代码在调用外部服务前持久化状态（数据库行、缓存、文件），服务失败时会发生什么？重试是否会产生重复？ | 用真实对象追踪失败路径。若状态在风险调用前创建，测试失败时的清理或重试的幂等性。 |
   | **还有哪些接口暴露了此功能？** Mixin、DSL、替代入口点（Agent vs Chat vs ChatMethods）。 | 在相关类中 grep 该方法/行为。若需要一致性，现在添加——而不是作为后续跟进。 |
   | **跨层错误策略是否一致？** 重试中间件 + 应用层回退 + 框架错误处理——它们是否冲突或导致双重执行？ | 列出每层的具体错误类。验证你的 rescue 列表与下层实际抛出的内容匹配。 |

   **何时跳过：** 叶子节点变更，无回调、无状态持久化、无并行接口。若变更是纯增量的（新辅助方法、新视图片段），检查只需 10 秒，结论是"无触发，跳过"。

   **何时最重要：** 任何触碰带回调的模型、带回退/重试的错误处理、或通过多接口暴露的功能的变更。


2. **增量提交**

   完成每个任务后，评估是否创建增量提交：

   | 提交时机... | 不提交时机... |
   |-------------|---------------|
   | 逻辑单元完成（模型、服务、组件） | 更大单元的一小部分 |
   | 测试通过 + 有实质性进展 | 测试失败 |
   | 即将切换上下文（后端 → 前端） | 纯脚手架，无行为 |
   | 即将尝试有风险/不确定的变更 | 需要写"WIP"提交消息 |

   **经验法则：** "我能写一条描述完整、有价值的变更的提交消息吗？能的话就提交。如果消息会是'WIP'或'partial X'，就等等。"

   若计划有 Implementation Units，以此作为提交边界的起始指引——但根据实现中的发现进行调整。一个单元可能比预期大需要多次提交，小的相关单元可能合为一次提交。使用每个单元的 Goal 来指导提交消息。

   **提交工作流：**
   ```bash
   # 1. 验证测试通过（使用项目的测试命令）
   # 示例：npm test, pytest, go test 等

   # 2. 仅暂存与此逻辑单元相关的文件（不要 `git add .`）
   git add <与此逻辑单元相关的文件>

   # 3. 使用约定式消息提交
   git commit -m "feat(scope): 此单元的描述"
   ```

   **处理合并冲突：** 若在 rebase 或 merge 时出现冲突，立即解决。增量提交使冲突解决更容易，因为每次提交都小而聚焦。

   **注意：** 增量提交使用简洁的约定式消息，不含署名页脚。最终阶段 4 的提交/PR 包含完整署名。

   **并行子代理模式：** 当单元作为并行子代理运行时，子代理不提交——编排器在整个并行批次完成后处理暂存和提交（见阶段 1 步骤 4 中的并行子代理约束）。本节的提交指导适用于内联和串行执行，以及编排器在并行批次完成后的提交决策。

3. **遵循已有模式**

   - 计划应引用相似代码——优先阅读这些文件
   - 精确匹配命名约定
   - 尽可能复用已有组件
   - 遵循项目编码标准（参见 AGENTS.md）
   - 有疑问时，grep 查找相似实现

4. **持续测试**

   - 每次重要变更后运行相关测试
   - 不要等到最后才测试
   - 立即修复失败
   - 为新行为添加新测试，为变更行为更新测试，为删除行为移除测试
   - **使用 mock 的单元测试证明逻辑在隔离环境中有效。使用真实对象的集成测试证明各层协同工作。** 若变更涉及回调、中间件或错误处理——你需要两者。

5. **持续简化**

   在完成一组相关实现单元后（或每 2-3 个单元），审查最近变更的文件寻找简化机会——合并重复模式、提取共享辅助函数、改善代码复用和效率。这在子代理模式下尤其有价值，因为每个代理使用隔离上下文，无法发现跨单元的模式。

   不要在每个单元后都简化——早期模式看似重复但可能在后续单元中刻意分化。等待自然的阶段边界或注意到累积复杂度时再进行。

6. **Figma 设计同步**（如适用）

   对于有 Figma 设计的 UI 工作：

   - 按照设计规范实现组件
   - 使用 figma-design-sync 代理迭代对比
   - 修复识别出的视觉差异
   - 重复直到实现匹配设计

7. **跟踪进度**
   - 完成任务时保持任务列表更新
   - 记录任何阻碍或意外发现
   - 若范围扩大则创建新任务
   - 向用户通报主要里程碑

### 阶段 3-4：质量检查与交付

当所有阶段 2 任务完成、执行转入质量检查时，阅读 `references/shipping-workflow.md` 获取完整交付工作流：质量检查、代码审查、最终验证、PR 创建和通知。

## 核心原则

### 快速启动，快速执行

- 开始时澄清一次，然后执行
- 不要等待完美理解——提问然后推进
- 目标是**完成功能**，而非创建完美流程

### 计划是你的向导

- 工作文档应引用相似代码和模式
   - 加载这些引用并遵循
   - 不要重新发明——匹配已有的

### 持续测试

   - 每次变更后运行测试，而非最后
   - 立即修复失败
   - 持续测试防止大惊喜

### 质量内建

   - 遵循已有模式
   - 为新代码编写测试
   - 推送前运行 lint
   - 审查每项变更——简单增量工作用内联审查，其他情况用完整审查

### 交付完整功能

   - 转到下一步前标记所有任务完成
   - 不要留下 80% 完成的功能
   - 一个交付的完整功能胜过一个完美但未交付的功能

## 常见陷阱

   - **分析瘫痪** —— 不要过度思考，阅读计划然后执行
   - **跳过澄清问题** —— 现在问，而不是构建错误的东西之后
   - **忽略计划引用** —— 计划中的链接是有原因的
   - **最后才测试** —— 持续测试，否则后续吃苦
   - **忘记跟踪进度** —— 随时更新任务状态，否则会忘记已完成的内容
   - **80% 完成综合症** —— 完成功能，不要提前离开
   - **跳过审查** —— 每项变更都要审查；只是深度不同
