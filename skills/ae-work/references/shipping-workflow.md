# 交付工作流

本文件包含交付工作流（阶段 3-4）。仅在所有阶段 2 任务完成、执行转入质量检查时加载。

## 阶段 3：质量检查

1. **运行核心质量检查**

   提交前务必运行：

   ```bash
   # 运行完整测试套件（使用项目的测试命令）
   # 示例：npm test, pytest, go test 等

   # 运行 lint（参见 AGENTS.md）
   # 推送到 origin 前使用 lint 检查
   ```

2. **代码审查**（必需）

   每项变更在交付前都要审查。深度随变更风险等级调整，但审查本身从不跳过。

   **层级 2：完整审查（默认）**——除非明确满足层级 1 标准，否则必需。调用 `ae:review` 技能并设置 `mode:autofix`，运行专业审查代理、自动应用安全修复、并将残余工作呈现为待办项。当计划文件路径已知时，传递 `plan:<path>`。这是强制的默认选项——仅在确认以下每条标准后才使用层级 1。

   **层级 1：内联自审**——仅在以下**全部四条**标准为真时允许的更轻量替代方案。选择层级 1 前，明确说明哪些标准适用及其原因。若有任何标准不确定，使用层级 2。
   - 纯增量（仅新文件，未修改已有行为）
   - 单一关注点（一个技能、一个组件——非跨模块）
   - 遵循模式（实现模仿已有示例，无新颖逻辑）
   - 忠实于计划（无范围膨胀、延迟问题的解决方案无意外）

3. **最终验证**
   - 所有任务标记为已完成
   - 测试已覆盖——测试通过且新增/变更的行为有对应测试覆盖（或有明确的不需要测试的理由）
   - Lint 通过
   - 代码遵循已有模式
   - Figma 设计匹配（如适用）
   - 无控制台错误或警告
   - 若计划有 `Requirements Trace`，验证每个需求已被完成的工作满足
   - 若记录了任何 `Deferred to Implementation` 问题，确认在执行过程中已解决

4. **准备运维验证计划**（必需）
   - 在每次变更的 PR 描述中添加 `## 部署后监控与验证` 章节。
   - 包含具体内容：
     - 日志查询/搜索词
     - 需要关注的指标或仪表板
     - 预期的健康信号
     - 失败信号和回滚/缓解触发条件
     - 验证窗口和负责人
   - 若确实无生产/运行时影响，仍包含该章节，内容为：`无需额外运维监控`及一行原因说明。

## 阶段 4：交付

1. **准备证据上下文**

   注意已完成的工作是否具有可观察行为（UI 渲染、CLI 输出、带可运行示例的 API/库行为、生成的产物或工作流输出）。PR 创建流程会在存在可观察行为时询问是否捕获证据。

2. **更新计划状态**

   若输入文档有 YAML frontmatter 中的 `status` 字段，将其更新为 `completed`：
   ```
   status: active  ->  status: completed
   ```

3. **提交并创建 Pull Request**

   执行以下步骤完成提交、推送和 PR 创建：

   ```bash
   # 确保所有变更已暂存
   git add <相关文件>

   # 使用约定式消息提交
   git commit -m "feat(scope): 完整变更描述"

   # 推送到远程
   git push -u origin <branch-name>
   ```

   然后使用 `gh` 命令创建 PR：
   ```bash
   gh pr create --title "PR 标题" --body "$(cat <<'EOF'
   ## 摘要
   <变更的要点总结>

   ## 测试说明
   <新增/修改的测试，执行的手动测试>

   ## 部署后监控与验证
   <见阶段 3 步骤 4>
   EOF
   )"
   ```

   在 PR 描述中提供：
   - 计划的摘要和关键决策
   - 测试说明（新增/修改的测试、执行的手动测试）
   - 步骤 1 中的证据上下文
   - Figma 设计链接（如适用）
   - 部署后监控与验证章节（见阶段 3 步骤 4）

   若用户倾向于只提交不创建 PR，仅执行 git commit 和 push，跳过 `gh pr create`。

4. **通知用户**
   - 总结完成的工作
   - 链接到 PR（如已创建）
   - 注明任何需要的后续工作
   - 如适用，建议后续步骤

## 质量检查清单

创建 PR 前，验证：

- [ ] 所有澄清问题已提出并回答
- [ ] 所有任务标记为已完成
- [ ] 测试已覆盖——测试通过且新增/变更的行为有对应测试覆盖（或有明确的不需要测试的理由）
- [ ] Lint 通过
- [ ] 代码遵循已有模式
- [ ] Figma 设计匹配实现（如适用）
- [ ] 证据决策在变更具有可观察行为时已处理
- [ ] 提交消息遵循约定式格式
- [ ] PR 描述包含部署后监控与验证章节（或明确的无影响说明）
- [ ] 代码审查已完成（内联自审或完整 `ae:review`）
- [ ] PR 描述包含摘要、测试说明和证据（如已捕获）

## 代码审查层级

每项变更都要审查。层级决定深度，而非是否审查。

**层级 2（完整审查）**——必需的默认选项。当计划路径可用时调用 `ae:review mode:autofix` 并传递 `plan:<path>`。安全修复自动应用；残余工作呈现为待办。除非明确确认全部四条层级 1 标准，否则始终使用此层级。

**层级 1（内联自审）**——仅在以下全部四条为真时允许（选择前逐条明确说明）：
- 纯增量（仅新文件，未修改已有行为）
- 单一关注点（一个技能、一个组件——非跨模块）
- 遵循模式（模仿已有示例，无新颖逻辑）
- 忠实于计划（无范围膨胀、延迟问题的解决方案无意外）
