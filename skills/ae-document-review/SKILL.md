# 文档审查

通过多角色分析审查需求或计划文档。并行调度专业审查代理，自动修复质量问题，并向用户展示需要决策的战略性问题。

## 阶段 0：检测模式

检查技能参数中的 `mode:headless`。参数可能包含文档路径、`mode:headless` 或两者兼有。以 `mode:` 开头的标记是标志，不是文件路径——从参数中移除它们，将剩余标记（如果有）作为阶段 1 的文档路径。

如果存在 `mode:headless`，则为后续工作流设置**无头模式**。

**无头模式**改变的是交互模型，而非分类边界。document-review 仍然应用相同的判断标准来区分什么有明确正确的修复方案、什么需要用户判断。唯一的区别在于非自动发现的处理方式：
- `auto` 修复静默应用（与交互模式相同）
- `present` 发现以结构化文本返回给调用方处理——不使用 question 工具提示，不需要交互式审批
- 阶段 5 立即返回"审查完成"（不提供优化/完成问题）

调用方收到的发现保留其原始分类，由调用方决定如何处理。

调用方通过在技能参数中包含 `mode:headless` 来启用无头模式，例如：
```
Skill("ae:document-review", "mode:headless docs/plans/my-plan.md")
```

如果不存在 `mode:headless`，技能以默认的交互模式运行，行为不变。

## 阶段 1：获取并分析文档

**如果提供了文档路径：** 读取文档，然后继续。

**如果未指定文档（交互模式）：** 询问要审查哪个文档，或使用文件搜索/Glob 工具在 `docs/brainstorms/` 或 `docs/plans/` 中查找最近的文档。

**如果未指定文档（无头模式）：** 输出"审查失败：无头模式需要文档路径。请使用以下方式重新调用：Skill(\"ae:document-review\", \"mode:headless <path>\")"，不调度代理。

### 分类文档类型

读取后，对文档进行分类：
- **requirements**——来自 `docs/brainstorms/`，关注构建什么和为什么构建
- **plan**——来自 `docs/plans/`，关注如何构建以及实现细节

### 选择条件角色

分析文档内容以确定要激活哪些条件角色。检查以下信号：

**product-lens**——当文档对构建什么和为什么构建做出可质疑的主张，或当提议的工作具有超越当前问题的战略影响力时激活。系统的用户可能是最终用户、开发者、运维人员、维护者或任何其他受众——审查标准不因领域而异。检查以下任一条件：

*条件 1——前提主张：* 文档对构建什么或为什么构建表明了立场，有经验的相关方可能合理地质疑——而不仅仅是描述任务或重述已知需求：
- 所述需求的问题框架不是不言自明的，或存在争议
- 可能存在替代方案的方案选择（隐式或显式）
- 明确排列构建优先级与推迟项的优先级决策
- 预测特定用户结果的目标陈述，而非仅仅重述约束或描述交付物

*条件 2——战略影响力：* 拟议的工作可能影响系统走向、用户认知或竞争定位，即使前提是合理的：
- 影响系统认知或品牌特征的变化
- 影响采纳、上手或认知负担的复杂性或简洁性押注
- 影响未来方向可行性的工作（路径依赖、架构承诺）
- 机会成本影响——构建这个意味着不构建别的

**design-lens**——当文档包含以下内容时激活：
- UI/UX 相关内容、前端组件或视觉设计语言
- 用户流程、线框图、屏幕/页面/视图提及
- 交互描述（表单、按钮、导航、弹窗）
- 响应式行为或无障碍相关引用

**security-lens**——当文档包含以下内容时激活：
- 认证/授权相关内容、登录流程、会话管理
- 暴露给外部客户端的 API 端点
- 数据处理、个人隐私信息、支付、令牌、凭证、加密
- 具有信任边界影响的第三方集成

**scope-guardian**——当文档包含以下内容时激活：
- 多个优先级层级（P0/P1/P2，必须有/应该有/可以有）
- 大量需求（>8 个独立需求或实现单元）
- 弹性目标、锦上添花项或"未来工作"章节
- 与既定目标不一致的范围边界语言
- 与需求没有明确关联的目标

**adversarial**——当文档包含以下内容时激活：
- 超过 5 个独立需求或实现单元
- 带有理由说明的明确架构或范围决策
- 高风险领域（认证、支付、数据迁移、外部集成）
- 新抽象、框架或重要架构模式的提议

## 阶段 2：公告并调度角色

### 公告审查团队

告知用户哪些角色将参与审查及其原因。对于条件角色，包含理由：

```
审查团队：
- coherence-reviewer（始终启用）
- feasibility-reviewer（始终启用）
- scope-guardian-reviewer -- 计划包含跨 3 个优先级层级的 12 个需求
- security-lens-reviewer -- 计划新增带认证流程的 API 端点
```

### 构建代理列表

始终包含：
- `coherence-reviewer`
- `feasibility-reviewer`

添加已激活的条件角色：
- `product-lens-reviewer`
- `design-lens-reviewer`
- `security-lens-reviewer`
- `scope-guardian-reviewer`
- `adversarial-document-reviewer`

### 调度

使用平台的 Task 工具**并行**调度所有代理。省略 `mode` 参数，使用用户配置的权限设置。每个代理接收根据下方子代理模板构建的提示，填入以下变量：

| 变量 | 值 |
|------|-----|
| `{persona_file}` | 代理 markdown 文件的完整内容 |
| `{schema}` | 下方包含的发现 schema 内容 |
| `{document_type}` | 阶段 1 分类的"requirements"或"plan" |
| `{document_path}` | 文档路径 |
| `{document_content}` | 文档完整文本 |

向每个代理传递**完整文档**——不要按章节拆分。

**错误处理：** 如果代理失败或超时，使用已完成代理的发现继续。在覆盖范围部分注明失败的代理。不要因单个代理失败而阻塞整个审查。

**调度限制：** 即使在最大规模（7 个代理）时也使用并行调度。这些是文档审查代理，范围有限，只读取单个文档——并行是安全且高效的。

## 阶段 3-5：综合、展示和下一步操作

所有调度的代理返回后，阅读 `references/synthesis-and-presentation.md` 了解综合流水线（验证、门控、去重、提升、解决矛盾、按自动修复类别路由）、自动修复应用、发现展示和下一步操作菜单。不要在代理调度完成之前加载此文件。
