# 阶段 3-5：综合、展示和下一步操作

## 阶段 3：综合发现

通过以下流水线处理所有代理的发现。**顺序很重要**——每一步依赖于前一步。

### 3.1 验证

根据 findings schema 检查每个代理返回的 JSON：
- 丢弃缺少 schema 中定义的任何必填字段的发现
- 丢弃具有无效枚举值的发现
- 在覆盖范围部分记录任何格式不正确的输出的代理名称

### 3.2 信心门控

抑制信心低于 0.50 的发现。将它们存储为遗留风险，以便在步骤 3.4 中可能提升。

### 3.3 去重

使用 `normalize(section) + normalize(title)` 对每个发现生成指纹。归一化：小写、去除标点、合并空白。

当指纹跨角色匹配时：
- 如果发现建议**相反的操作**（例如一个说删除，另一个说保留），不合并——保留两者以在 3.5 中进行矛盾解决
- 否则合并：保留最高严重级别，保留最高信心，合并所有证据数组，记录所有同意的审查者（例如"coherence, feasibility"）
- **覆盖范围归属：** 将合并后的发现归属于信心最高的角色。减少其他角色的发现计数*以及*对应的路由桶（自动或待处理），使 `发现 = 自动 + 待处理` 保持精确。

### 3.4 提升遗留风险

扫描遗留风险（在 3.2 中抑制的发现），查找：
- **跨角色佐证**：角色 A 的遗留风险与角色 B 的阈值以上发现重叠。以 P2 提升，信心 0.55-0.65。从佐证的阈值以上发现继承 `finding_type`。
- **具体阻塞风险**：遗留风险描述了一个会阻塞实现的具体风险。以 P2 提升，信心 0.55。设置 `finding_type: omission`（作为遗留风险出现的阻塞风险本质上都是关于文档未能解决的问题）。

### 3.5 解决矛盾

当角色对同一章节有不同意见时：
- 创建一个**组合发现**，呈现双方观点
- 设置 `autofix_class: present`
- 设置 `finding_type: error`（矛盾本质上是关于文档所说的冲突内容，而非遗漏的内容）
- 以权衡的方式呈现，而非裁决

具体冲突模式：
- coherence 说"保留以保持一致性" + scope-guardian 说"删除以保持简洁" -> 组合发现，让用户决定
- feasibility 说"这不可能" + product-lens 说"这是必要的" -> P1 发现，以权衡方式呈现
- 多个角色标记同一问题 -> 合并为单个发现，注明共识，提高信心

### 3.6 提升模式解析的发现

扫描 `present` 发现中的代码库模式解析的 auto 资格。当**三个条件全部满足**时，将 `present` 提升为 `auto`：

1. 发现的 `why_it_matters` 引用了具体的现有代码库模式——不仅是"最佳实践"或"惯例"，而是具有文件、函数或使用引用的具体模式
2. 发现包含遵循该引用模式的具体 `suggested_fix`
3. 不存在真正的权衡——代码库上下文解决了使用哪种方法的歧义

原则：当审查者提到多种理论方法，但代码库已有明确的既定模式使一种方法明显正确时，代码库上下文解决了问题。如果证据显示代码库已经做出选择，那么偶尔提到的替代方案不会构成真正的权衡。

额外的自动提升模式（在以下情况将 `present` 提升为 `auto`）：
- 发现识别出文档中事实上不正确的行为，且建议的修复描述了正确的行为（不是设计方案之间的选择）
- 发现识别出缺少行业标准安全控制，且文档自身上下文使遗漏明显是错误的（不是对所描述系统的合理设计选择），建议的修复遵循既定实践
- 发现识别出不完整的技术描述，且完整版本可直接从代码库推导（审查者引用了具体代码显示描述应该说什么）

如果发现涉及范围或优先级变更，而文档作者可能权衡了审查者不可见的权衡，则不要提升。

### 3.7 按自动修复类别路由

**严重性和 autofix_class 是独立的。** 如果正确的修复方案很明显，P1 发现也可以是 `auto`。判断标准不是"有多重要？"而是"是否有一个明确正确的修复方案，还是需要判断？"

| 自动修复类别 | 路由 |
|-------------|------|
| `auto` | 自动应用——一个明确正确的修复方案。包括内部一致性修正（某部分权威于另一部分）、由文档自身内容机械推导的补充、以及代码库模式解析的修复。 |
| `present` | 单独呈现供用户判断 |

将任何缺少 `suggested_fix` 的 `auto` 发现降级为 `present`。

**符合自动修复条件的模式：** 摘要/细节不匹配（正文权威于概述）、错误计数、可从文档其他部分推导的缺失列表条目、过时的内部交叉引用、术语漂移、正文/图表矛盾（正文更详细时以正文为准）、由其他内容机械推导的缺失步骤、由周围上下文推导的未说明阈值、正确补充明显的完整性缺口、审查者引用具体现有模式且 suggested_fix 遵循该模式的代码库模式解析修复、正确行为从上下文或代码库显而易见的事实上不正确的行为、缺少有已知实现的标准安全控制、完整版本可从代码库推导的不完整技术描述。如果修复需要对*做什么*做出判断（而不仅仅是*写什么*）且代码库上下文未解决歧义，则应归入 `present`。

### 3.8 排序

按以下顺序排列发现以展示：P0 -> P1 -> P2 -> P3，然后按发现类型（错误在前，遗漏在后），然后按信心（降序），然后按文档顺序（章节位置）。

## 阶段 4：应用和展示

### 应用自动修复

在**单次操作**中将所有 `auto` 发现应用到文档：
- 使用平台的编辑工具内联编辑文档
- 跟踪已更改的内容以用于"已应用的自动修复"章节
- 不请求审批——这些有明确正确的修复方案

在输出摘要中列出每个自动修复，以便用户了解更改内容。每项修复使用足够细节以传达实质（章节、更改内容、审查者归属）。这对于涉及内容添加或影响文档含义的修复尤为重要——用户不应需要 diff 文档才能理解审查做了什么。

### 展示剩余发现

**无头模式：** 不使用交互式提问工具。将所有非自动发现输出为结构化文本摘要，供调用方解析和处理：

```
文档审查完成（无头模式）。

已应用 N 个自动修复：
- <章节>：<更改内容>（<审查者>）
- <章节>：<更改内容>（<审查者>）

发现（需要判断）：

[P0] 章节：<章节> — <标题>（<审查者>，信心 <N>）
  原因：<why_it_matters>
  建议修复：<suggested_fix 或"无">

[P1] 章节：<章节> — <标题>（<审查者>，信心 <N>）
  原因：<why_it_matters>
  建议修复：<suggested_fix 或"无">

遗留风险：
- <风险>（<来源>）

推迟问题：
- <问题>（<来源>）
```

省略任何零项的章节。然后直接进入阶段 5（在无头模式下立即返回）。

**交互模式：**

使用审查输出模板展示 `present` 发现（阅读 `references/review-output-template.md`）。在每个严重级别内，按类型分离发现：
- **错误**（设计张力、矛盾、不正确的陈述）在前——这些需要解决
- **遗漏**（缺失步骤、缺省细节、遗漏条目）在后——这些需要补充

顶部简要摘要："已应用 N 个自动修复。K 个发现待处理（X 个错误，Y 个遗漏）。"

包含覆盖范围表格、已应用的自动修复、遗留风险和推迟问题。

### 受保护的产物

在综合过程中，丢弃任何建议删除或移除以下目录中文件的发现：
- `docs/brainstorms/`
- `docs/plans/`
- `docs/solutions/`

这些是流水线产物，不应被标记为待删除。

## 阶段 5：下一步操作

**无头模式：** 立即返回"审查完成"。不提问。调用方接收阶段 4 的文本摘要并处理任何剩余发现。

**交互模式：**

**使用平台的交互式提问工具提问**——不要将问题作为纯文本输出打印：
- question 工具
- 回退（无提问工具可用时）：展示编号选项并停止；等待用户的下一条消息

提供以下两个选项。使用阶段 1 的文档类型设置"审查完成"的描述：

1. **再次优化**——处理上述发现，然后重新审查
2. **审查完成**——基于文档类型的描述：
   - 需求文档："使用 ae:plan 创建技术计划"
   - 计划文档："使用 ae:work 执行实现"

经过 2 次优化迭代后，建议完成——收益递减的可能性较大。但如果用户希望继续，允许继续。

返回"审查完成"作为调用方的终止信号。

## 禁止事项

- 不要重写整个文档
- 不要添加用户未讨论过的新章节或新需求
- 不要过度工程化或增加复杂性
- 不要创建单独的审查文件或添加元数据章节
- 不要修改调用方技能（ae:brainstorm、ae:plan 或调用 document-review 的外部插件技能）

## 迭代指引

在后续迭代中，重新调度角色并重新综合。自动修复机制和信心门控会阻止已修复的相同发现再次出现。如果发现跨迭代重复，建议完成。
