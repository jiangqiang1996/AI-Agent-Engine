# 会话交接指令（ae-handoff）需求

## 问题框架
用户需要在opencode中实现类似handoff的会话交接功能，主动触发后新开独立会话，并自动将当前会话的重要结论代入新会话，避免重复沟通上下文，提升多阶段任务的执行效率。

## 需求

**核心功能**
- R1. 提供用户主动触发命令 `/ae-handoff`，执行后自动创建新的独立会话
- R2. 自动分析当前会话的全部内容，提取核心结论、已做决策、待办事项、关键上下文等重要信息
- R3. 将提取的重要结论自动加入新会话的系统提示词中，作为新会话的上下文基础
- R4. 新会话为完全独立会话，不保留原会话的历史聊天记录

## 成功标准
- 用户输入 `/ae-handoff` 命令后10秒内完成会话创建和上下文带入
- 提取的重要结论准确率不低于90%（定义：会话中应提取的关键信息中至少90%被成功提取，且提取内容中错误信息占比不超过10%）
- 无关键信息遗漏（定义：关键信息指会话中产生的核心结论、已达成决策、待办事项、项目上下文等影响后续任务执行的信息，遗漏判定以用户反馈为准）
- 新会话能够正确识别带入的上下文，不需要用户重复告知之前的结论

## 范围边界
- 暂不支持自动触发会话交接
- 暂不支持用户手动选择或编辑要带入的结论
- 暂不支持新会话和原会话的关联回溯
- 暂不支持额外参数配置（如指定代理、模型、自定义提取规则等）

## 关键决策
- 触发方式选择用户主动触发：降低复杂度，符合用户预期，避免意外创建会话
- 结论提取选择自动提取：减少用户操作成本，提升使用体验
- 新会话选择独立会话：保持会话上下文干净，避免原会话冗余信息干扰
- 结论呈现选择系统提示词一部分：保证上下文默认生效，避免用户误修改
- 暂不支持额外参数：优先交付核心功能，降低首次实现的复杂度

## 依赖 / 假设
- 假设opencode平台支持通过工具/命令创建新会话
- 假设opencode平台支持自定义新会话的系统提示词

- [影响R1][技术] 如何调用opencode平台API创建新会话，需要探索平台支持的能力
- [影响R3][技术] 如何调用opencode平台API自定义新会话的系统提示词，需要探索平台支持的能力
