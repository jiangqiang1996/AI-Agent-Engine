# 迭代任务循环技能迁移需求

> 由 `/ae-lfg` → `ae:brainstorm` 生成，日期：2026-04-19

## Problem Frame

源项目 QianYi 的 `.opencode/commands/task-loop.md` 实现了一个"执行→验证→判定"循环命令，用于反复派发子代理执行任务并自动验证，直到目标达成或退出条件触发。

现需将其迁移到 AE 项目，统一为 `ae` 前缀命名，并**完全重构**而非照搬。重构的核心目标：

1. **融入 AE 技能生态** — 与已有的 `ae:work`、`ae:review`、`ae:plan` 等技能互补而非重叠
2. **适配 AE 技术栈** — 验证手段从 Java/Maven 切换到 TypeScript/Node.js
3. **增强能力** — 结合 AE 已有的模式（子代理派发、计划文件格式、技能链路）提升完善度

## 现有技能定位分析

### ae:work vs ae:task-loop 定位差异

| 维度 | ae:work | ae:task-loop（待迁移） |
|------|---------|----------------------|
| 驱动方式 | 计划文档驱动（结构化计划→逐步执行） | 目标驱动（一句话目标或计划文件→循环执行） |
| 执行模式 | 线性推进，按优先级逐个完成 | 循环推进，每轮执行→验证→判定 |
| 验证方式 | 系统级测试检查 + 增量提交 | 独立验证子代理，逐项判定通过/不通过 |
| 失败处理 | 修复后继续 | 修复后重新验证，连续相同失败触发瓶颈分析 |
| 适用场景 | 有明确计划的结构化实现 | 目标明确但路径不确定的探索性任务 |
| 输入粒度 | 完整计划文档 | 一句话描述或计划文件路径 |

**结论**：两者互补。`ae:work` 适合"知道怎么做"的场景，`ae:task-loop` 适合"知道要什么但不确定怎么做"的场景。

### 与 ae:review 的关系

`ae:review` 是代码审查技能，专注于 diff 级别的质量审查。`ae:task-loop` 的验证步骤更轻量——关注"目标是否达成"而非"代码是否规范"。两者验证目标不同，不重叠。

### 与 ae:plan 的关系

`ae:task-loop` 支持计划文件模式（传入计划路径，每轮推进一个步骤），此时与 `ae:plan` 产生的计划文件格式天然集成。

## Requirements

### R1. 命令与技能注册

**作为** AE 用户，
**我希望** 通过 `/ae-task-loop` 命令调用迭代任务循环，
**以便** 快速启动一个"执行直到完成"的工作循环。

验收标准：
- 创建 `commands/ae-task-loop.md` 命令入口（thin wrapper 模式，与现有命令一致）
- 创建 `skills/ae-task-loop/SKILL.md` 技能定义（完整逻辑）
- frontmatter 包含 `name: ae:task-loop`、`description`、`argument-hint`
- 构建后自动同步到 `.opencode/commands/` 和 `.opencode/plugins/`

### R2. 目标解析与完成标准推导

**作为** AE 用户，
**我希望** 输入一句话目标或计划文件路径，系统自动推导完成标准，
**以便** 无需手动编写详细的验证条件。

验收标准：
- 支持两种输入模式：一句话描述 / 计划文件路径（自动检测）
- 计划文件模式：解析 AE 计划格式（frontmatter + 实现单元），提取每个单元的目标和验证标准
- 一句话描述模式：自动将目标拆解为可判定的完成条件
- 隐含条件推导：根据任务类型自动追加编译通过、测试通过等隐含条件（适配 AE 技术栈）
- 向用户展示完整验证标准，等待确认后才开始执行

### R3. 循环执行管线

**作为** AE 用户，
**我希望** 系统在循环中自动执行任务并验证结果，
**以便** 我只需关注最终目标而无需逐步监督。

验收标准：
- 三步循环管线：执行（子代理）→ 验证（子代理）→ 判定
- 执行步骤：必须通过 Task 工具启动 `general` 类型子代理
- 验证步骤：必须通过 Task 工具启动独立的 `general` 类型子代理
- 判定步骤：主会话根据子代理返回做一次判断
- 每轮有实质性操作，禁止空转
- 计划文件模式：每轮只推进当前待执行的下一个实现单元，不跳步不并行

### R4. 执行子代理 prompt 结构化

**作为** 技能设计者，
**我希望** 执行子代理接收结构化的 prompt，
**以便** 子代理有足够的上下文高效完成任务。

验收标准：
- prompt 包含：背景、本轮目标、完成条件、输出要求四部分
- 背景包含：已完成步骤、前几轮失败方案摘要、当前轮次
- 输出要求子代理返回三项：本轮动作、变更清单、执行结论
- 允许子代理调用其他能力（bash、edit、read 等）完成具体任务

### R5. 验证子代理 prompt 结构化

**作为** 技能设计者，
**我希望** 验证子代理独立验证每项完成条件，
**以便** 验证结果客观可追溯。

验收标准：
- prompt 包含：验证目标、本轮变更、验证要求三部分
- 逐项检查每个完成条件，给出"通过"/"不通过"判定
- 不通过时必须给出具体失败信息（错误输出、缺失内容等）
- 禁止主观判断（如"看起来没问题"）
- 返回三项：验证结论、逐项结果、失败摘要

### R6. 验证手段适配 AE 技术栈

**作为** AE 项目开发者，
**我希望** 验证手段匹配本项目的实际技术栈，
**以便** 验证结果真实可靠。

验收标准：
- 验证手段表覆盖 AE 项目实际使用的构建和测试方式
- TypeScript/Node.js 项目：`npm run build`、`npm run test`
- 文件创建/修改：读取目标文件，逐项检查关键内容
- 计划文件执行：逐项比对计划步骤与实际文件内容
- 验证手段根据项目类型自动选择

### R7. 退出条件与判定逻辑

**作为** AE 用户，
**我希望** 循环在合理时机退出，
**以便** 不会无限循环也不会过早放弃。

验收标准：
- 全部完成条件满足 → 输出完成报告，标记 DONE
- 连续两轮验证不通过且不通过项完全相同 → 停止循环，输出瓶颈分析
- 达到最大轮次 → 停止循环，输出进展与未完成原因
- 用户回复终止语义 → 立即停止，输出当前状态摘要
- 验证不通过但未达退出条件 → 向用户报告本轮结果和差距，等待确认后继续

### R8. 每轮输出与退出输出

**作为** AE 用户，
**我希望** 每轮看到清晰的执行和验证摘要，
**以便** 了解当前进展和差距。

验收标准：
- 每轮输出三项：本轮动作、验证输出、差距
- 退出时输出：当前状态摘要、总迭代轮次、关键变更清单
- 输出格式清晰、结构化

### R9. 隐含条件推导（AE 技术栈适配）

**作为** 技能设计者，
**我希望** 系统根据任务类型自动追加合理的隐含完成条件，
**以便** 验证更全面，减少遗漏。

验收标准：
- 推导表覆盖 AE 项目实际场景（TypeScript/Node.js 而非 Java/Maven）
- 包含但不限于：代码实现、编译通过、测试通过、计划执行、通用任务
- 用户提供了验证条件时仍然执行推导，未覆盖的条件追加到验证标准中

### R10. 执行约束

**作为** AE 用户，
**我希望** 循环执行有合理的安全约束，
**以便** 不会发生意外的破坏性操作。

验收标准：
- 默认最大轮次 10，不自作主张超过
- 破坏性操作（删除文件等）需用户确认
- 验证全通过时直接 DONE 结束，不额外等待确认
- 验证不通过且未达退出条件时，等待用户确认后才进入下一轮

## Success Criteria

1. `ae-task-loop` 可通过 `/ae-task-loop <目标>` 或 `/ae-task-loop <计划文件路径>` 启动
2. 自动推导完成标准并向用户确认
3. 循环执行→验证→判定，每轮有清晰输出
4. 所有退出条件（完成、瓶颈、轮次上限、用户终止）正确触发
5. 验证手段匹配 AE 技术栈（TypeScript/Node.js）
6. 与 `ae:work`、`ae:plan`、`ae:review` 定位清晰互补
7. 命令和技能遵循 AE 项目命名和结构规范

## Scope Boundaries

### In Scope

- `commands/ae-task-loop.md` 命令入口
- `skills/ae-task-loop/SKILL.md` 技能定义
- 构建同步（自动复制到 `.opencode/commands/`）
- 循环执行管线的完整逻辑
- AE 技术栈适配的验证手段
- 隐含条件推导表
- 执行约束和退出条件

### Out of Scope

- 不修改任何已有技能（`ae:work`、`ae:plan`、`ae:review` 等）
- 不添加新的子代理类型（使用已有的 `general` 类型）
- 不涉及插件源码修改（仅命令和技能文件）
- 不涉及数据库操作或外部服务集成

## Key Decisions

### KD1. 命令+技能双层架构

**决策**：采用与现有 AE 命令一致的模式——命令为 thin wrapper，技能承载完整逻辑。

**理由**：保持项目一致性。所有 `ae-*` 命令都遵循此模式（如 `ae-work.md` → `ae:work`）。

### KD2. 验证手段从 Java/Maven 切换到 TypeScript/Node.js

**决策**：完全重写验证手段表，匹配 AE 项目实际技术栈。

**理由**：源项目的验证手段面向 Spring Boot/Java 项目（`mvn compile`、`mvn test`），AE 项目使用 TypeScript/Node.js（`npm run build`、`npm run test`）。

### KD3. 与 ae:work 互补而非替代

**决策**：`ae:task-loop` 定位为"目标驱动的迭代执行器"，与 `ae:work` 的"计划驱动的线性执行器"互补。

**理由**：`ae:work` 擅长从结构化计划逐步实现，`ae:task-loop` 擅长从简单目标反复尝试直到成功。两种模式都有价值。

### KD4. 保持子代理派发模式

**决策**：保留源项目的"执行子代理 + 验证子代理"双子代理派发模式。

**理由**：opencode 的 Task 工具支持 `general` 类型子代理，此模式在两个平台都适用，且验证独立性更好。

### KD5. 增强 prompt 结构化

**决策**：保留源项目的 prompt 结构化设计（背景/目标/条件/输出要求），并增加与 AE 技能生态的衔接信息。

**理由**：结构化 prompt 是 task-loop 的核心价值之一，确保子代理输出可解析的结果。
