# 阶段 3-5：综合、展示和下一步操作

## 阶段 3：综合发现

### 3.1 验证

根据 findings schema 检查每个代理返回的 JSON。丢弃缺少必填字段的发现。记录格式不正确的代理。

### 3.2 信心门控

抑制信心低于 0.50 的发现。将它们存储为遗留风险。

### 3.3 去重

使用 `normalize(section) + normalize(title)` 生成指纹。

当指纹跨角色匹配时：
- 如果发现建议**相反的操作**，不合并——保留两者
- 否则合并：保留最高严重级别和信心，合并证据，记录所有同意的审查者
- 归属于信心最高的角色，减少其他角色的计数

### 3.4 提升遗留风险

- **跨角色佐证**：角色 A 的遗留风险与角色 B 的发现重叠 → P2 提升
- **具体阻塞风险**：会阻塞实现的风险 → P2 提升

### 3.5 解决矛盾

创建**组合发现**，呈现双方观点。设置 `autofix_class: present`、`finding_type: error`。

### 3.6 提升模式解析的发现

当三个条件全部满足时将 `present` 提升为 `auto`：
1. 引用了具体的现有代码库模式
2. 包含遵循该模式的 `suggested_fix`
3. 不存在真正的权衡

### 3.7 按自动修复类别路由

| 类别 | 路由 |
|------|------|
| `auto` | 自动应用——明确正确的修复方案 |
| `present` | 单独呈现供用户判断 |

将缺少 `suggested_fix` 的 `auto` 发现降级为 `present`。

### 3.8 排序

P0 → P1 → P2 → P3，然后错误在前遗漏在后，然后信心降序，然后文档顺序。

## 阶段 4：应用和展示

### 应用自动修复

在**单次操作**中将所有 `auto` 发现应用到文档。不请求审批。

### 展示剩余发现

**无头模式：** 将所有非自动发现输出为结构化文本摘要：

```
文档审查完成（无头模式）。

已应用 N 个自动修复：
- <章节>：<更改内容>（<审查者>）

发现（需要判断）：

[P0] 章节：<章节> — <标题>（<审查者>，信心 <N>）
  原因：<why_it_matters>
  建议修复：<suggested_fix 或"无">

遗留风险：
- <风险>（<来源>）

推迟问题：
- <问题>（<来源>）
```

**交互模式：** 使用审查输出模板展示 `present` 发现（阅读 `references/review-output-template.md`）。在每个严重级别内按类型分离。

### 受保护的产物

丢弃任何建议删除以下目录中文件的发现：
- `docs/ae/brainstorms/`
- `docs/ae/plans/`
- `docs/ae/solutions/`

## 阶段 5：下一步操作

**无头模式：** 立即返回"审查完成"。不提问。

**交互模式：** 提供两个选项：
1. **再次优化** — 处理发现后重新审查
2. **审查完成** — 进入下一步

经过 2 次优化迭代后，建议完成。

返回"审查完成"作为调用方的终止信号。

## 禁止事项

- 不要重写整个文档
- 不要添加用户未讨论过的新章节
- 不要创建单独的审查文件或添加元数据章节

## 迭代指引

后续迭代中重新调度角色。如果发现跨迭代重复，建议完成。
