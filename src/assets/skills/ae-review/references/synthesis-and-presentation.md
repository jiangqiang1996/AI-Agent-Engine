# 阶段 5-7：综合、展示和审查后

## 阶段 5：合并发现结果

### 5.1 验证

根据 findings schema 检查每个代理返回的 JSON。丢弃缺少必填字段的发现。记录格式不正确的代理。

### 5.2 置信度门控

抑制低于 0.60 的发现（P0 在 0.50+ 保留）。

### 5.3 去重

使用 `normalize(file) + normalize(title)` 生成指纹。

当指纹跨角色匹配时：
- 如果发现建议**相反的操作**，不合并——保留两者
- 否则合并：保留最高严重级别和置信度，合并证据，记录所有同意的审查者
- 归属于置信度最高的角色，减少其他角色的计数

### 5.4 跨审查者一致

2+ 审查者标记同一问题时提高置信度 0.10。

### 5.5 分离预先存在问题

标记 `pre_existing: true` 的发现不计入审查结论，单独展示。

### 5.6 解决分歧

创建**组合发现**，呈现双方观点。保留最保守的路由。

### 5.7 合并文档审查发现

将阶段 4b 中 ae-document-review 返回的文档发现合并到统一报告中。注意：需求文档和计划文档默认排除——除非用户明确指定纳入，此时它们会经由 ae-document-review 审查后在此合并。
- 文档发现使用 ae-document-review 的 findings schema（section、finding_type、autofix_class: auto/present）
- 代码发现使用代码审查的 findings schema（file、line、autofix_class: safe_auto/gated_auto/manual/advisory）
- 两种发现的严重级别（P0-P3）和置信度可直接对比排序
- 文档发现不参与代码审查的 autofix_class 路由（不转换为 safe_auto/gated_auto）

### 5.8 规范化路由

出现分歧时选择更保守的路由。（仅适用于代码审查发现。）

### 5.9 划分工作（代码审查发现）

| 队列 | 包含 | 动作 |
|------|------|------|
| 修复队列 | `safe_auto` | 自动应用 |
| 剩余可操作队列 | `gated_auto`、`manual` | 需批准或设计决策 |
| 只读队列 | `advisory` | 仅展示 |

将缺少 `suggested_fix` 的 `safe_auto` 发现降级为 `gated_auto`。

### 5.10 排序

严重级别 → 置信度 → 文件路径 → 行号。

## 阶段 6：合成并展示

使用审查输出模板（`references/review-output-template.md`）中管道分隔的 markdown 表格展示发现。包含：头部、发现表格、需求完整性、已应用修复、剩余工作、预存问题、经验教训、代理原生缺口、覆盖范围、结论。

### 模式驱动的展示

- **交互模式** — 自动应用 `safe_auto`，策略问题询问用户
- **自动修复模式** — 仅应用 `safe_auto`，保留其余
- **只读模式** — 不构建修复队列，仅展示
- **无头模式** — 单轮 `safe_auto`，结构化文本输出

### 受保护产物

丢弃任何建议删除以下目录中文件的发现：
- `docs/ae/brainstorms/`
- `docs/ae/plans/`
- `docs/ae/solutions/`
- `.opencode/`

## 阶段 7：审查后

### 步骤 1：应用修复

使用一个修复器和 `max_rounds: 2` 应用修复。

### 步骤 2：写入运行产物

写入 `docs/ae/review/<run-id>/`，包含 metadata.json。

### 步骤 3：最终下一步（仅交互模式）

- PR 模式：推送修复 / 退出
- 分支模式：创建 PR（推荐）/ 继续 / 退出
- 默认分支：继续 / 退出

### 步骤 4：更新状态文件

更新 `.opencode/review-state.json`（将当前 HEAD 写入 `lastReviewed`）。

## 质量门

1. 每个发现都是可操作的
2. 没有因未仔细阅读代码导致的误报
3. 严重级别校准正确
4. 行号准确（适用时）
5. 受保护产物得到尊重
6. 发现不重复 linter 输出

## 禁止事项

- 不要重写整个文件来修复一个小问题
- 不要添加用户未讨论过的新功能
- 不要创建单独的审查文件或添加元数据章节到源文件
