# AE 规范保存

项目规范管理助手，严格按以下流程执行。

**重要约束**：
1. 规范只能保存为**项目级规范**，写入 `.opencode/rules/` 目录。禁止保存到全局目录
2. 项目规范的范围覆盖 `opencode.json` 中 `instructions` 数组引用的所有文件（包括 glob 匹配、明确路径、`@` 前缀引用的文件）

## 第一步：识别候选规范

回顾当前会话，识别值得持久化的通用约定。关注以下两类信号：

1. **用户的纠正与强调** — 用户明确纠正了做法、强调的偏好、要求遵循的约束
2. **反复出现的模式** — 本次会话中多次出现且被用户默认认可的约定

涵盖领域：
- 编码约定（命名、风格、模式）
- 架构决策（模块划分、依赖关系、分层策略）
- 工作流规范（测试策略、Git 工作流、部署流程）
- 工具配置（构建工具、lint 规则、CI/CD）
- 业务规则（领域约束、业务逻辑约定）
- API 设计（接口风格、错误处理、版本策略）

### 通用性过滤

每条内容必须**同时满足**以下条件：

- 适用于**整个项目**而非某次特定任务
- **未来遇到同类场景时都应遵循**
- 是**通用的、可复用的**约定
- 表述为**规则或原则**

**必须排除**：仅适用于特定函数/文件的做法、具体的配置值或硬编码路径、一次性调试步骤、框架默认行为、不具备可复用性的实现细节。

### 已有规范去重

在进入第二步之前，必须先读取 `.opencode/rules/` 目录下所有已有文件的内容，将候选规范与已有规范逐条比对：

- **完全重复**：候选规范与已有规范表述的规则完全一致或仅措辞不同但语义相同 → **直接排除**，告知用户该规范已存在，无需重复保存
- **已有规范覆盖更广**：已有规范的规则范围包含候选规范 → **直接排除**，告知用户已有规范已涵盖此约定
- **候选规范是对已有规范的细化补充**：保留候选规范，标注为"细化补充"，写入时追加到已有章节或作为新章节
- **候选规范与已有规范完全矛盾**：保留候选规范，标注为"⚠️ 矛盾"，必须重点提醒用户

去重后如果没有剩余的候选规范，告知用户"所有候选规范均已存在于项目规范中，无需保存"并停止。

## 第二步：会话历史交叉验证

此步骤内联执行会话历史验证逻辑，验证候选规范的历史维度。解决三个问题：

1. **反复性** — 该约定是否在多个历史会话/产物中反复出现？（反复出现 = 更值得固化）
2. **归入检测** — 已有领域文件中是否存在相关内容？（互补 = 追加，冲突 = 需谨慎）
3. **决策背景** — 历史上是否有相关讨论？（已被否决的约定 = 必须警告用户）

### 验证执行方式

使用 Task 工具以子代理方式验证，将所有候选规范的关键词一次性传入：

```
请验证以下候选规范的历史背景：

1. {候选规范 1 的核心关键词}
2. {候选规范 2 的核心关键词}
...

搜索范围：
- 项目规范文件（.opencode/rules/ 目录下所有 .md 文件）
- opencode.json 中 instructions 引用的所有文件
- 项目产物（docs/ 目录下的文件）
- Git 历史（近期提交信息）

对每条候选规范，返回：反复性判定、已有规范状态、决策上下文。
```

### 会话历史内省

同时回顾当前会话的完整对话历史，提取以下信息辅助验证：

- **决策上下文** — 当前会话中关于候选规范的讨论背景、做出决策的原因
- **反复出现验证** — 相关话题在本次会话中出现的次数和一致性
- **用户明确态度** — 用户是否对相关约定表达过明确赞同或否定

### 已有规范合并检查

读取 `.opencode/rules/` 目录下所有已有文件，检查候选规范应归入的领域文件是否已存在：

- **领域文件已存在**：标注为"追加"，指明目标文件路径和将追加的章节标题
- **领域文件不存在**：标注为"新建"，指明将创建的文件路径
- **⚠️ 矛盾**：候选规范与已有文件中的某个章节存在**完全矛盾的逻辑**（如"禁止 X" vs "必须 X"）→ 标注为"⚠️ 矛盾追加"，**必须重点提醒用户**，展示矛盾对比：
  ```
  ⚠️ 矛盾检测
  
  已有规范：{已有章节标题}
  已有内容：{已有规范的关键规则摘要}
  候选规范：{候选规范的关键规则摘要}
  矛盾点：{具体矛盾描述}
  
  请选择：
  1. 覆盖已有规范（候选规范将替换已有章节）
  2. 保留已有规范（放弃候选规范）
  3. 两者都保留（作为并列章节，需手动标注适用场景）
  ```

### 信号标注

| 信号 | 含义 | 处理方式 |
|------|------|---------|
| 反复出现 | 历史中多次出现相关约定 | **强烈推荐**保存 |
| 首次出现 | 历史中未找到相关记录 | 推荐，标注为"新约定" |
| 已有互补 | 对应领域文件已存在且内容互补 | 追加到已有文件 |
| 细化补充 | 候选规范是对已有规范的细化 | 追加到已有文件，标注细化关系 |
| ⚠️ 矛盾 | 候选规范与已有规范逻辑完全矛盾 | **必须重点提醒用户**，用户选择覆盖则替换已有章节 |
| 曾被否决 | 历史中有类似讨论但被明确否决 | **必须警告用户**，展示否决理由 |

## 第三步：展示并确认

向用户展示带有历史信号的候选规范列表：

```
发现以下可能值得保存的项目规范（共 N 条）

━━━ 规范 1 ━━━
领域分类：{分类名称} → {文件路径}
写入方式：{新建/追加/冲突追加}
历史信号：{信号}
内容：
{规范完整内容}

━━━ 规范 N ━━━
...

请选择：
1. 输入要保存的编号（如 "1,3"）
2. 输入 "全部" 保存所有规范
3. 输入 "取消" 放弃保存
4. 对某条规范提出修改意见（如 "2: 把 XX 改为 YY"）
```

对 `⚠️ 矛盾` 的规范，必须用醒目格式展示矛盾对比，并提供覆盖/保留/两者都保留三个选项。对 `曾被否决` 的规范，必须额外展示历史否决理由。

## 第四步：前置检查

读取 `opencode.json`，检查 `instructions` 数组中是否包含 `.opencode/rules/**/*.md`：

- **已存在**：继续
- **不存在**：提示用户确认是否自动添加；拒绝则停止并提示手动添加

## 第五步：写入文件

### 5.1 确定领域分类

每条候选规范必须归入以下领域分类之一：

| 分类 | 子目录 | 文件名 | 涵盖内容 |
|------|--------|--------|---------|
| 核心规范 | `core` | `base.md` | 项目总览、语言要求、核心原则 |
| 代码风格 | `core` | `code-style.md` | 命名、格式、TypeScript/Effect/Zod 规范 |
| 架构设计 | `architecture` | `architecture.md` | 模块划分、分层原则、依赖方向、错误处理 |
| 测试规范 | `core` | `testing.md` | 测试框架、覆盖率、命名、Mock 规范 |
| Git 工作流 | `core` | `git-workflow.md` | 分支策略、提交规范、禁止事项 |
| Agent 设计 | `core` | `agent-design.md` | 工具设计、Agent 组织、Hook 规范、Prompt 工程 |
| 前端规范 | `architecture` | `frontend.md` | UI 组件、样式体系、交互模式、TUI 约定 |
| 后端规范 | `architecture` | `backend.md` | API 设计、数据层、服务层约定 |
| 数据库规范 | `architecture` | `database.md` | SQL 约定、迁移策略、ORM 使用 |
| 配置规范 | `architecture` | `config.md` | 环境配置、构建工具、CI/CD |
| 运维规范 | `architecture` | `ops.md` | 部署流程、监控、日志约定 |

**原则**：
- 文件名使用**通用、总结性**的领域名称，而非具体规则的名称
- 同一领域下的所有规范归入同一文件，通过二级标题分隔不同条目
- 当候选规范不属于上述分类时，可新增分类，但文件名仍须使用通用领域名称

### 5.2 合并优先策略

**核心原则：合并优先，新建其次。**

1. 读取对应领域分类的已有文件（路径：`.opencode/rules/{子目录}/{文件名}.md`）
2. 判断结果：
   - **文件已存在，候选规范为细化补充**：将候选规范追加到已有文件的末尾，作为新的二级标题章节。更新文件头部的"最后更新"日期
   - **文件已存在，候选规范与已有章节⚠️矛盾，用户选择覆盖**：用候选规范替换已有章节的内容，更新文件头部的"最后更新"日期
   - **文件已存在，候选规范与已有章节⚠️矛盾，用户选择两者都保留**：将候选规范追加为新章节，在两个章节中分别标注适用场景
   - **文件不存在**：创建新文件，写入规范内容
3. 禁止为单条规范创建独立文件。每条规范必须归入其领域分类的通用文件中

### 5.3 写入格式

**新建文件格式**：

```markdown
# {领域名称}规范

> 本文件由 `/ae-save-rules` 命令生成，最后更新：{当前日期}

## {规范标题}

{规范内容}
```

**追加到已有文件格式**：

在已有文件末尾追加：

```markdown

## {规范标题}

{规范内容}
```

同时更新文件头部"最后更新"日期为当前日期。

### 完成提示

```
规范已保存

{对每条规范列出：}
- {规范标题} → {保存路径} {新建文件/追加到已有文件/更新已有章节}
```
