# 迭代任务循环

目标驱动的迭代执行器。**本技能不编写计划文档，不产出设计产物，只做一件事：循环执行直到目标达成或确认无法达成。**

**适用场景：** 探索性调试、环境搭建、遗留代码修复

**不适用场景：** 有明确计划 → `ae:work`；需求不明确 → `ae:brainstorm`；代码审查 → `ae:review`

## 核心原则

1. **交互全部前置** — 所有需要用户参与的问题在 Phase 0 解决，Phase 0 结束后直至 Phase 3 之前，**禁止使用 `question` 工具或以任何形式向用户提问**。子代理同样禁止提问。
2. **成功条件锁死** — Phase 0 确认的成功条件列表不可在循环中修改。如果发现条件本身有误，那是瓶颈退出，不是修改条件的理由。
3. **原始目标为真** — 首次执行时将用户原始目标文本原封不动传入子代理，由子代理自行解读执行。
4. **验证先行，判定统一** — 每轮循环必须先验证再判定。所有退出判定（全部通过/瓶颈/上限/REBUILD/FIX）在 Step 2 中统一决策，不在其他位置检查退出条件。
5. **遇到歧义自行决策** — 循环体中遇到任何不确定的情况，自行选择最优解，不得因此暂停或退出。
6. **状态持续追踪** — 跨轮次维护：轮次计数、瓶颈计数（连续无新增通过项的轮数）、历史通过条件集合。

## 输入

<goal> #$ARGUMENTS </goal>

**如果 `<goal>` 为空，询问用户：** "你想完成什么目标？"

支持目标前缀技能名（如 `ae:work 修复登录bug`），提取技能名作为执行技能注入。

## 工作流

### Phase 0：准备（允许交互）

本阶段解决所有用户问题。主会话仅保留：目标文本、各子会话返回的摘要、用户回答。

#### 0.1 解析用户目标（主会话）

解读 `<goal>`，识别核心意图、是否包含技能前缀、约束条件。

#### 0.2 扫描项目状态（子会话）

委派子代理扫描项目，判断目标相关实现是否存在。子代理应扫描相关文件/代码/配置，返回摘要：**是否有实现**（是/否）、**实现概况**（简短描述）。

#### 0.3 预分析提问（子会话）

委派子代理分析目标歧义和交互节点。子代理应：扫描目标涉及的 SKILL.md 提取交互节点，分析首次执行/验证/修复中可能需要的用户决策，将问题分为不可默认项和可默认项（附带建议默认值）。

如果目标明确无歧义且无需用户决策，结果可为空。

#### 0.4 收集用户回答（主会话）

使用 `question` 工具逐条向用户确认 0.3 中**不可默认的项**。可默认项直接采用建议默认值，不询问用户。

#### 0.5 推导成功条件（子会话）

委派子代理根据目标、项目状态、用户回答推导可验证的成功条件。每条条件必须可客观验证（文件存在、代码行为、测试结果等），数量控制在 3-8 条，格式为 `SC-N: 描述`。

#### 0.6 展示确认清单 + 用户确认（主会话）

向用户展示完整清单：目标解读、项目状态、已预处理的交互节点及答案、成功条件列表。等待用户确认或修改。

用户确认后，**锁死成功条件和目标解读，此后不可修改。**

> **Phase 0 结束。⛔ 从现在起直到 Phase 3，禁止任何形式的用户交互。**

### Phase 1：首次执行（条件性）

根据 0.2 的扫描结果：

- **无实现** → 委派子代理执行首次实现。将用户原始目标文本原封不动传入，子代理自行解读执行。可使用任意工具/技能/代理，遇到歧义自行决策。
- **已有实现** → 跳过，直接进入 Phase 2。

### Phase 2：循环验证/修复（禁止交互）

**参数：** 上限 30 轮，瓶颈阈值连续 3 轮无新增通过项。

每轮严格按 Step 1 → Step 2 → Step 3 顺序执行。

#### Step 1：验证（子会话）

轮次 +1。委派子代理逐条对照成功条件验证当前项目状态。子代理对每条给出通过/不通过判定，附具体证据（文件路径、代码片段或测试结果）。

#### Step 2：判定（主会话）

先更新追踪状态：对比本轮验证结果与历史通过记录，计算新增通过的条件。有新增则瓶颈计数归零，无新增则瓶颈计数 +1。

然后**严格按以下优先级依次检查，命中第一条即执行对应动作，不再检查后续**：

| 优先级 | 条件 | 动作 |
|--------|------|------|
| 1 | 全部条件通过 | 退出循环 → Phase 3（全部通过 ✅） |
| 2 | 差距较大或方向有误（多数条件未通过，或反复修复同一问题无效） | 标记 REBUILD → Step 3 |
| 3 | 有具体可定位的未通过项 | 记录问题清单，标记 FIX → Step 3 |
| 4 | 瓶颈计数 ≥ 3（连续 3 轮无新增通过项） | 退出循环 → Phase 3（瓶颈退出 ⚠️） |
| 5 | 轮次 ≥ 30 | 退出循环 → Phase 3（上限退出 ⚠️） |

#### Step 3：执行修复（子会话）

根据 Step 2 标记委派子代理：

- **REBUILD**：较大范围重做相关实现，不要在失败基础上打补丁。传入未通过条件列表、目标参考、成功条件。
- **FIX**：针对问题清单逐一定向修复。传入未通过条件的具体原因和位置、成功条件。

两种模式均：可使用任意工具/技能/代理，遇到歧义自行决策，禁止提问，完成后报告影响了哪些文件。

#### → 回到 Step 1

### Phase 3：退出处理（允许交互）

根据退出原因输出结果：

**全部通过 ✅** — 输出 DONE，逐条列出每条成功条件的通过证据。

**瓶颈退出 ⚠️** — 展示诊断摘要（总轮次、通过/未通过条件、可能原因），使用 `question` 工具提供选项：继续尝试 / 结束任务 / 切换其他方案。

**上限退出 ⚠️** — 展示进度摘要（总轮次 30、通过/未通过条件、最近 3 轮变化趋势），使用 `question` 工具提供选项：增加轮次 / 结束任务 / 切换其他方案。

若用户希望继续，可再次调用本技能并附加补充说明。
