# 工作执行技能

按计划高效执行工作，保持质量并完成功能交付。

## 简介

本技能接收一份工作文档或一段描述工作的提示词，并系统化地执行。核心目标是**交付完整功能**。

## 输入文档

<input_document> #$ARGUMENTS </input_document>

## 执行工作流

### 阶段 0：输入分流

**计划文档**（输入为路径）→ 跳至阶段 1。

**裸提示词**（输入为工作描述）：

1. **扫描工作区域** — 识别可能变更的文件、查找测试文件、记录本地模式
2. **评估复杂度并路由：**

| 复杂度 | 信号 | 操作 |
|--------|------|------|
| **简单** | 1-2 个文件，无行为变更 | 直接实现，无需任务列表 |
| **小/中** | 范围明确，约 10 文件以内 | 构建任务列表，进入阶段 1 |
| **大** | 跨模块、架构决策、10+ 文件 | 建议先运行 `/ae-plan` |

---

### 阶段 1：快速启动

1. **阅读计划并澄清**（若从裸提示词进入则跳过）
   - 完整阅读工作文档，视为决策产物
   - 检查每个单元的 `Execution note`、`Deferred to Implementation`、`Scope Boundaries`
   - 阅读计划中引用的文件
   - 若用户明确要求 TDD，即使计划无 Execution note 也要遵循
   - 若有不明之处立即提问

2. **准备环境**
   - 检查当前分支
   - 若在默认分支上：创建新分支或使用 worktree
   - 若已在功能分支：建议重命名无意义分支名
   - 未经明确许可不提交到默认分支

3. **创建待办列表**（若已构建或路由为简单则跳过）
   - 从计划推导任务，包含依赖关系和优先级排序
   - 将每个单元的 Verification 作为完成信号

4. **选择执行策略**

| 策略 | 适用场景 |
|------|----------|
| **内联** | 1-2 个小任务 |
| **串行子代理** | 3+ 有依赖的任务 |
| **并行子代理** | 3+ 通过安全检查的任务 |

**并行安全检查：** 构建文件到单元的映射，发现重叠则降级为串行。

**并行子代理约束：** 指示每个子代理不暂存、不提交、不运行测试套件。

### 阶段 2：执行

1. **任务执行循环**

   按优先级处理每个任务。遵循 Execution note 指引。

   **测试发现** — 变更前查找已有测试文件。新行为添加新测试、变更行为修改测试、删除行为移除测试。

   **测试场景完整性** — 编写测试前检查计划的测试场景是否覆盖所有适用类别：

| 类别 | 适用时机 |
|------|----------|
| 正常路径 | 始终 |
| 边界情况 | 有意义的边界 |
| 错误/失败路径 | 有失败模式 |
| 集成 | 跨层 |

   **系统级测试检查** — 任务完成前检查：回调/中间件、集成测试覆盖、失败时的孤立状态、跨接口一致性、错误策略。

2. **增量提交**

   逻辑单元完成且测试通过时提交。不使用"WIP"消息。使用约定式格式。

3. **遵循已有模式** — 优先阅读计划引用的文件，精确匹配命名约定。

4. **持续测试** — 每次重要变更后运行测试。

5. **持续简化** — 每 2-3 个单元后审查简化机会。

6. **Figma 设计同步**（如适用）— 按照设计规范实现，迭代对比直到匹配。

7. **跟踪进度** — 保持任务列表更新，记录阻碍和发现。

### 阶段 3-4：质量检查与交付

当所有阶段 2 任务完成时，阅读 `references/shipping-workflow.md` 获取完整交付工作流。

## 核心原则

- **快速启动，快速执行** — 澄清一次，然后执行
- **计划是向导** — 遵循已有模式和引用
- **持续测试** — 每次变更后测试，非最后
- **质量内建** — 遵循模式、编写测试、推送前 lint
- **交付完整功能** — 标记所有任务完成，不留 80% 功能
