# 需求捕获

此内容在阶段 3 开始时加载——在协作对话产生了值得保留的持久决策之后。

---

此文档应表现为没有 PRD 仪式的轻量级 PRD。包含规划需要良好执行的内容，跳过对范围没有价值的章节。

需求数据文档用于产品定义和范围控制。**不要**包含实现细节（库、Schema、端点、文件布局），除非头脑风暴本身是技术性的且这些细节是决策的主题。

**非简单工作的必需内容：**
- 问题框架
- 带有稳定 ID 的具体需求或预期行为
- 范围边界
- 成功标准

**在有实质帮助时包含：**
- 关键决策及其理由
- 依赖或假设
- 待定问题
- 考虑过的替代方案

**文档结构：** 使用此模板并省略不适用的可选章节：

```markdown
---
date: YYYY-MM-DD
topic: <kebab-case-topic>
---

# <主题标题>

## 问题框架
[谁受影响，什么在变化，为什么重要]

## 需求

**[分组标题]**
- R1. [具体需求]
- R2. [具体需求]

**[分组标题]**
- R3. [具体需求]

## 成功标准
- [如何知道这解决了正确的问题]

## 范围边界
- [明确的非目标或排除项]

## 关键决策
- [决策]: [理由]

## 依赖 / 假设
- [仅在有意义时包含]

## 待定问题

### 规划前需解决
- [影响 R1][用户决策] [规划前必须回答的问题]

### 推迟到规划
- [影响 R2][技术] [应在规划期间回答的问题]

## 下一步
[如果规划前需解决为空：-> /ae-plan]
[如果规划前需解决不为空：-> 恢复 /ae-brainstorm]
```

**视觉沟通** — 当需求加上视觉辅助后会显著更容易理解时，包含一个。阅读 `references/visual-communication.md` 获取决策标准和放置规则。

对于**标准**和**深入**头脑风暴，通常需要一份需求数据文档。

对于**轻量**头脑风暴，保持文档紧凑。当用户只需要简短对齐时，跳过文档创建。

对于只有 1-3 个简单需求的非常小的需求数据文档，普通要点需求是可以接受的。对于**标准**和**深入**需求数据文档，使用稳定 ID（`R1`、`R2`、`R3`），以便规划和后续审查可以无歧义地引用。

当需求跨越多个不同关注点时，在需求部分的加粗主题标题下分组。分组依据是逻辑主题的不同，而非需求数量。需求保留其原始稳定 ID——编号不按组重新开始。

当工作简单时，合并章节而不是填充。简短的需求数据文档好于臃肿的。

在最终确定之前，检查：
- 如果这个头脑风暴现在结束，`ae:plan` 还需要发明什么？
- 是否有任何需求依赖于声称在范围之外的东西？
- 是否有任何未解决的项目实际上是产品决策而非规划问题？
- 是否有不应该出现的实现细节泄漏了进来？
- 是否有任何需求声称基础设施不存在而该声明尚未对照代码库验证？
- 是否有一个低成本的变更会使这显著更有用？
- 视觉辅助是否能帮助读者比纯文字更快地理解需求？

如果规划需要发明产品行为、范围边界或成功标准，头脑风暴还没有完成。

在写入之前确保 `docs/ae/brainstorms/` 目录存在。

如果文档包含待定问题：
- 仅对真正阻塞规划的问题使用 `规划前需解决`
- 当技术问题在规划中更好回答时，将其放在 `推迟到规划` 下
- 显式地向前传递推迟的问题，而不是将它们视为失败
