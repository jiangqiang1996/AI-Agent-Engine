# 深化工作流

本文件包含信心检查执行路径（5.3.3-5.3.7）。仅在深化门控确定有必要时加载。

## 5.3.3 评分信心差距

使用检查清单优先、风险加权的评分轮次。

对于每个部分，计算触发计数 + 风险加分 + 关键部分加分。候选条件：2+ 总分，或高风险领域中 1+ 分且实质性重要。

仅选择得分最高的 2-5 个部分。

**部分检查清单：**

- **需求追溯** — 需求模糊、成功标准缺失、单元未追溯需求
- **上下文与研究** — 模式被命名但未用于决策、高风险缺乏外部支撑
- **关键技术决策** — 决策无理由、未解释权衡
- **未解决问题** — 产品阻塞项被隐藏为假设、推迟项过于模糊
- **高层技术设计** — 媒介不当、包含实现代码、标注缺失
- **实现单元** — 依赖顺序不清晰、文件路径缺失、测试场景模糊
- **系统范围影响** — 受影响接口缺失、失败传播不足
- **风险与依赖** — 风险无缓解、推出影响缺失

## 5.3.4 报告和分发

报告正在加强哪些部分。每个部分最多 1-3 个代理，总共通常不超过 8 个。

**固定的部分到代理映射：**

- 需求/未解决问题 → `spec-flow-analyzer`、`repo-research-analyst`
- 上下文/研究空白 → `learnings-researcher`、`framework-docs-researcher`
- 关键技术决策 → `architecture-strategist`
- 实现单元 → `repo-research-analyst`、`pattern-recognition-specialist`
- 系统范围影响 → `architecture-strategist` + 风险匹配的专家
- 风险/运维 → 安全/数据/部署/性能专家

## 5.3.5 选择研究执行模式

使用能工作的最轻量模式：直接模式（默认）或制品暂存模式（仅大范围）。

## 5.3.6 运行研究

并行启动选定的代理。优先使用本地仓库证据。

## 5.3.6b 交互式发现审查（仅交互模式）

自动模式跳过此步骤。交互模式中逐个呈现每个代理的发现，用户选择接受/拒绝/讨论。

如果用户没有接受任何发现，跳过文档审查和综合。

## 5.3.7 综合并更新计划

仅加强选定的部分。允许的更改包括：澄清决策理由、加强追溯、重新排序单元、添加缺失模式引用、扩展风险处理。

**不要**：添加实现代码、git 命令、重写整个计划、发明新的产品需求。
